# XA-Guard Helm chart

This chart deploys four workloads: XA-Guard API, resilience worker, reference
business API, and web console. It also renders Services, Ingress, a migration
Job, ConfigMap, PodDisruptionBudgets, HPAs, and a release-scoped NetworkPolicy.

Production is the default profile. PostgreSQL, OIDC, and KMS are external, and
secret values are never generated by the chart. Before install, provision:

- `global.externalSecrets.runtime` with `database-url`, `oidc-client-secret`,
  `oidc-introspection-client-secret`, `kek-keyring`, and
  `internal-authorization-key` (names are configurable);
- `global.externalSecrets.businessApi` with `database-url` and `api-key`;
- approved immutable application image digests in each `image.digest` field;
- the real OIDC discovery/JWKS URLs and approved external endpoint CIDRs.

`referenceInfra.enabled=true` is an interface for kind/reference values. It does
not install bundled Keycloak, PostgreSQL, or KMS resources.

```bash
helm lint deploy/helm/xa-guard
helm template xa-guard deploy/helm/xa-guard --namespace xa-guard
helm upgrade --install xa-guard deploy/helm/xa-guard \
  --namespace xa-guard --create-namespace -f values.production.yaml
```

The default NetworkPolicy denies external egress. Kubernetes NetworkPolicy
cannot select DNS names, so provide only approved endpoint CIDRs through
`networkPolicy.externalEgressCidrs`, use the reference-only override, or enforce
FQDN policy with the cluster CNI. An empty list is intentionally fail closed.

The migration Job is a normal revision-scoped resource. Helm creates its
ConfigMap and ServiceAccount first; API and worker readiness must stay closed
until the schema is current.
