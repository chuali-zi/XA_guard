# XA-Guard Helm chart 0.2.0

This chart deploys XA-Guard API, resilience Worker, reference business API,
Console/BFF, a revision-scoped migration Job, Services, Ingress, PDBs, optional
HPAs, and dependency-scoped NetworkPolicies. Production remains the default:
OIDC, PostgreSQL, and the HTTP key provider are external and no credentials are
generated by Helm.

## Required external Secrets

The default HTTP key-provider mode reads:

- `global.externalSecrets.runtime`: `database-url`, `oidc-client-secret`,
  `oidc-introspection-client-secret`, and `internal-authorization-key`;
- `global.externalSecrets.businessApi`: `database-url` and `api-key`;
- `global.externalSecrets.keyProvider`: `auth-token`;
- optional CA Secret named by `global.kms.caSecretName`, with the key selected
  by `global.kms.caSecretKey`.

HTTP mode does not reference or mount `kek-keyring`. Setting
`global.kms.provider=local` changes the contract: workloads read only
`kek-keyring` from the runtime Secret and do not read the HTTP endpoint, token,
or CA. Local mode is intended for explicitly approved development profiles.

## HA scheduling and shutdown

API and Worker default to two replicas. Both use required pod anti-affinity and
`DoNotSchedule` topology spreading across `kubernetes.io/hostname`. Operators
can make anti-affinity preferred or change topology keys through each
component's `scheduling` values. PDBs, pre-stop drain delays, and termination
grace periods are independently configurable. Worker exposes port 8082 with
`/livez`, `/readyz`, and `/metrics`; readiness is supplied by the Worker runtime,
not a sidecar.

HPAs are available but disabled by default. Before enabling one, provide enough
topology domains for its maximum replica count or change required anti-affinity
to preferred; otherwise the scheduler will correctly leave excess replicas
pending.

## Network policy

The chart creates a namespace default-deny policy plus one policy for each of
API, Worker, business API, Console, and migration. Internal dependencies use
pod selectors and exact ports. External dependencies have separate CIDR/port
sets:

- `networkPolicy.dependencies.oidc`;
- `networkPolicy.dependencies.postgres`;
- `networkPolicy.dependencies.keyProvider`.

All external CIDR lists are empty by default, intentionally leaving production
pods unready until an operator supplies approved addresses. Standard Kubernetes
NetworkPolicy cannot select FQDNs; use maintained endpoint CIDRs or an approved
CNI FQDN policy. Never replace these lists with `0.0.0.0/0`.

## Render and install

```bash
helm lint deploy/helm/xa-guard --strict
helm template xa-guard deploy/helm/xa-guard --namespace xa-guard
helm upgrade --install xa-guard deploy/helm/xa-guard \
  --namespace xa-guard --create-namespace -f values.production.yaml \
  --wait --wait-for-jobs
```

Use immutable image digests for production. `referenceInfra.enabled=true` only
marks a reference profile and enables explicitly configured HTTP allowlists; it
does not install Keycloak, PostgreSQL, or the key provider. The Kind profile in
`deploy/kind/` runs those dependencies outside the cluster.

The migration Job is named by Helm revision, or by `migration.nameSuffix` for
an explicit idempotence rerun. API and Worker `/readyz` remain closed when the
binary schema, PostgreSQL, or required downstream readiness checks fail.
