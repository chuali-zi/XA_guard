# 拓展路线 — 一天/世界应该怎么长大

> 层级：参考蓝图。本文直接回答"**应该拓展什么**"：从当前竖切到"真实政企一天"的成长路径。
> 拓展的第一原则仍是 PRD §6：**加场景 = 加数据，不改内核**；拓展面来自内核通用，不来自一次建很多东西。

## 1. 拓展的五个维度

| 维度 | 从 | 到 |
|---|---|---|
| **域广度** | Office + Business Data 竖切 | 6 域全接入并并发运转 |
| **席位深度** | 单 Live seat | 多域多级（L1–L4）多 seat，按域分配 |
| **业务并发** | 顺序事件列表 | 业务时钟 + 队列 + 审批状态 + 跨域依赖 + 重试 + 并发 |
| **注入面** | 仅 `mailbox:` | mailbox/rag/log/plugin/ticket/supply/insider/meeting/policy 全谱 |
| **追责/报告** | 单次 verdict | 多 agent 委托链追责 + 反事实 A/B + live 统计 + 看板 |

## 2. 逐域接入顺序（与 SP2 对齐）

先竖切证明"内核通用"，再逐域拓宽，每接一个域都保留良性对照与 smoke 证据：

```
1) Office + Business Data   ← SP2 首个竖切（复用已跑通 office/mail + 数据域）
2) Operations（log: 注入面）
3) Dev Supply（plugin:/supply:/aibom: 注入面）
4) Governance（policy: 注入面、注册表、seat 生命周期）
5) Audit（AT10 审计完整性、replay/取证、export evidence）
```

判据（PropertyEngine）随域扩展新增属性族（越权、供应链声明漂移、审计断链…），仍是**场景绑定的数据**，不改内核判据机制。

## 3. 能力深化顺序（与 SP3–SP6 对齐）

| 阶段 | 拓展内容 | 对应 SP |
|---|---|---|
| 注入面通用化 | `mailbox:` → 多 scheme handler；组合投毒 | SP3 |
| 红队工作台 | CLI 看图→投毒→A/B→读证据→提 finding→固化；后置 Web UI | SP4 |
| 多 agent + 追责 | 多 seat/委托、身份/授权/委托三链、从坏状态点名元凶、反事实对照 | SP5 |
| 演示 + 统计 | 现场对照 demo、看板、ASR_null−ASR_guard、N≥2 置信区间、artifact 索引 | SP6 |

## 4. 世界拟真度拓展（让"一天"更像真实政企）

- **组织演进**：HR 入职/调岗/离职驱动 seat 生命周期（创建→分配→升级→冻结→销毁），审计连续性不断。
- **数据演进**：注册表、预算、合同状态随一天变化；次日以收工快照为基线。
- **信任边界丰富**：甲方政府客户、承包商、供应商、公网多类外部接收方，各自不同的外发规则。
- **成本模型**：按 seat 级别计预算归属与上限（模拟，与真实账单/比赛预算无关），可作为"资源滥用"类判据的素材。
- **失败与噪声**：告警去抖、重试、误报、队列积压——让攻击"混在正常业务里"，而不是被摆拍。

## 5. 不拓展什么（守住 YAGNI 与红线）

- 不做真实外发/真实凭据/生产数据/公网目标/真实破坏性命令。
- 不为扩题库而扩题库——平台价值在内核通用与红队自由，不在预置很多攻击。
- 不把 p0/p1/p2 benchmark 那套"做题/排行榜"重新引回来（那正是本轮要摆脱的漂移）。
- 内核不因某个域/某次演示而加场景专用分支。
