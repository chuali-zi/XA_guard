# Open Agent Range — 产品需求文档 (PRD)

> 状态：**已由作者审核并冻结（2026-07-03）。** 北极星已固定；如需修改须显式走一次评审。
> 这是本项目的北极星，只描述"为什么做 / 做成什么 / 什么感觉"，**不描述怎么实现**。
> 实现细节由各子项目(SP)自己的设计文档承担。本文档要薄，要自由。
> 作者一句话  完全核心！！！！“这个产品是一个模拟沙盘，不是普通的bench，不是普通的题目，而是真正的模拟一个真实的政企企业的一天，其中不要固化题目，不要写题目，就是纯纯的模拟一个正常的一个工作流程，一个真实的靶场，让红队可以完全自由的从任何角度，任何地方，都进行渗透”
---

## 1. 一句话

一个**开放的企业级智能体红队靶场平台**：搭一个尽量真实、每天照常运转的企业世界，把所有接口都敞开给红队从任意角度、任意组合投毒攻关，靠一本**不可篡改的地面真值账本**把"到底出了什么事、该谁负责"如实记下来——用来把真实的系统性问题**逼出来**，并检验 XA-Guard 这类防护能不能把"黑盒混乱"变成"可裁决、可追责"。

---

## 2. 为什么做（真正的问题）

政企不敢用 agent，**根本原因不是提示注入**——注入是可防的，防起来并不难。

根本原因是**乱**：黑盒、不可控、出了事查不清是谁干的、员工用起来太混乱，索性直接不用。

现有的 LLM benchmark 基本不可信，那是"做题"，逼不出真实的系统性失败。

我们要的不是做题，是**一个真实的场景，把问题逼出来**。

---

## 3. 这是什么

一个开放平台 = **一个通用内核 + 可无限扩展的场景**：

- 一个**仪表化的、尽量真实的企业世界**：人、角色、数据、工具、每天照常的行为流程、信任边界。
- 一本**不可篡改的地面真值账本**：如实记录每一次"谁、以什么身份、动了什么工具、碰了哪条数据、发给了谁"。这本账既是安全判据，也是追责证据。
- **所有接口对红队敞开**：可从任意角度、任意组合投毒——钓鱼、社会工程学、skill 投毒、MCP 投毒、内鬼员工、多链组合，以及**还没被想到的新招**。
- **被测防护(SUT，如 XA-Guard)在环裁决**；靶场根据副作用和账本判定输赢与追责，**而不是靠预设脚本**。

---

## 4. 灵魂：自由

这是本项目的**第一原则，高于一切**：

- **不脚本化攻击。** 攻击永远是红队的自由；靶场绝不把攻击写成固定 JSON 塞在固定位置。
- **不固化内核。** 内核永远不知道"这一题的机密是什么、攻击会怎么来"。什么算机密、谁有权、什么算出事，全部由**场景当数据声明**。
- **开放一切接口。** 现实里能被攻击的地方，这里都要能被攻击。
- **真实优先。** 世界尽量拟真，每天有照常的业务流程；攻击**混在正常业务里发生**，而不是被摆拍出来。
- **平台性来自内核通用，不来自一次建很多东西。**

---

## 5. "赢"的哲学（是判据，不是规则）

红队"赢" = 让世界进入一个**本不该出现的地面真值状态**：

- 机密数据到了无权的接收方，或
- 无权主体借 agent 之手越权完成了特权动作，或
- 某个有害后果**无法被追责**，或
- ……（可扩展，不封闭）

判据**从账本读出**，是通用属性，不是预设步骤；每个场景绑定自己的事实。攻击怎么来无所谓，只看世界有没有进入坏状态。

---

## 6. 内核 vs 场景（解耦契约的灵魂）

- **内核** = 通用物理引擎：世界模型、账本、全量仪表化、属性引擎、SUT/Seat 接口。**场景无关**。
- **场景** = 数据：一个世界实例 + 声明的事实（什么机密、谁有权）+ 敞开的注入面 + 绑定的属性。
- **加场景 = 加数据，不改内核。** 这是"开放"能免费长出来的原因。

---

## 7. 不做什么（尽量少）

- 不是一个跑分 benchmark，不追求排行榜数字。
- 不做真实外发、不用真实凭据/生产数据、不攻击公网目标。
- 内核不预置任何具体攻击、任何具体机密。

除此之外**尽量不设限**——设限是场景的事，不是平台的事。

---

## 8. 成功长什么样

能上台演一段**现场对照**：同一个真实场景，裸奔的 agent 制造了一次泄漏/混乱、事后你根本查不清是谁干的；换上 XA-Guard，同样的场景被拦下、账本上一条链清清楚楚指出元凶。评委一眼看懂"企业为什么不敢用、这个产品到底解决了什么"。

---

## 9. 范围与节奏（子项目群）

本平台按子项目分批交付，各自走 spec → plan → 实现：

```
SP1  内核 Kernel（通用物理引擎）      ← 先钉死
SP2  首个参考场景（垂直切片）          ← 证明内核真的通用可用
SP3  注入面通用化                      ← "多角度投毒"落地
SP4  红队工作台
SP5  多 agent + 追责深化
SP6  演示 / 看板
SP7  产品完成态验收（真实一天 / 自由注入 / live SUT / 判据族）
```

本 PRD 只固定"为什么 / 是什么 / 什么感觉"；各 SP 的"怎么做"由各自的设计文档承担。
SP7 不是新增题目或实现分支，而是把 SP1–SP6 汇总成"何时才算真的符合本 PRD"的验收总 spec。
