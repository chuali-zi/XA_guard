# 备选产品形态 · Skill 方案

> **文档定位**：产品形态可行性评估的**备选方案 #1**。当前主推 MCP Server 形态（详见 [产品架构.md](./产品架构.md)），本文评估"用 Skill 作为主形态或补充形态"的可行性。
>
> **结论先看**（client-friendly 摘要）：**作为补充形态可以加分；作为替代主形态不建议。**
>
> **详细调研报告**：[../docs-research/skill-form.md](../docs-research/skill-form.md)（约 4700 字 + 44 个引用来源）

---

## 1. Skill 是什么

**Skill = "配方"**（Anthropic 官方定位）。简单说：一份用自然语言写的"能力说明书"，存为 `SKILL.md` 文件，AI 读了就知道遇到某类任务时该做什么。

```
任务："清理服务器日志"
                ↓
AI 读取 SKILL.md（你装的"清理日志技能"）
                ↓
AI 按照 SKILL.md 描述的步骤执行
```

**与 MCP 的根本区别**（Anthropic 原话）：
> **"MCP 是水管，Skill 是配方"**
> MCP 让 AI 能调用外部工具，Skill 给 AI 提供"如何使用工具完成任务"的知识。

## 2. 时间线（业界进展）

| 时间 | 事件 |
|---|---|
| 2024-11 | MCP 协议首发 |
| 2025-10 | Anthropic Claude Skills 首发 |
| 2025-12 | Skill 开放标准 |
| 2026-02~03 | 企业治理特性补齐 |

Skill 比 MCP **晚一年**才热门起来——评委对 Skill 的熟悉度可能远不如 MCP。

---

## 3. 关键技术特性

### Skill 的优势
- **Token 开销极低**：比 MCP 工具描述少 **17-32 倍**（少传 prompt）
- **文件即分发**：一个 `SKILL.md` 文件就是一个 Skill，发布简单
- **可审计**：纯文本，可 Git 版本管理
- **跨平台**：Codex / Gemini CLI / Copilot / Kiro / TWG 等都开始支持 SKILL.md 规范

### Skill 的硬伤
- **不能主动调用外部资源**：Skill 是"知识"，不是"代码"。要做事必须借 MCP 或 bash + curl
- **无持久态**：每次对话重新加载，**不能做长时监控**
- **与"安全工具"角色冲突**（**这条是致命的**）

---

## 4. 致命问题：Skill 是 prompt injection 载体

学术界已经证明这一点：

| 论文 | 内容 |
|---|---|
| **arXiv 2510.26328**（ELLIS Tübingen, 2025-10） | 系统性证明 SKILL.md **天然是 prompt injection 载体** |
| **arXiv 2602.20867**（2026 早期） | 描述 **ClawHavoc 事件**：**1200 个恶意 Skill** 渗入 marketplace |

**逻辑悖论**：
> 我们要做的是 **AI 智能体安全防护工具**。
> Skill 这种"载体形式"本身已被证明**存在被 prompt injection 攻击的天然短板**。
>
> **用一个"载体本身不安全"的形态做"安全工具"——这是双战场**：既要做防护，又要先证明自己不被攻击。

---

## 5. 国产 IDE 覆盖率风险（也是致命的）

我们的项目主推国产 AI 工具（Trae / CodeBuddy / 通义灵码）。但 Skill 在国产生态的覆盖率（截至 2026-05）：

| 国产 IDE | MCP 原生支持 | Skill (SKILL.md) 原生支持 |
|---|---|---|
| **Trae**（字节） | ✅ | ❌ |
| **通义灵码**（阿里） | ✅ | ❌ |
| **文心快码**（百度） | ✅ | ❌ |
| **CodeBuddy**（腾讯云） | ✅ | ⚠ 部分（自己的 Rules 系统，格式不同） |
| 智谱清言桌面版 | 渐进式 | ❌ |

**对比**：MCP 在国产生态**全面覆盖**；Skill 在国产生态**几乎为零**。

**这与我们项目"主推国产"的叙事直接冲突**——如果转 Skill 主形态，**演示视频拍不出来**（Trae 不支持 Skill）。

## 6. 假想做 `xa-guard` Skill

如果硬要做，工程估算：

| 项 | 工程量 |
|---|---|
| `xa-guard` Skill（SKILL.md + 配套脚本）| 15-20 人天 |
| 同等 MCP Server | 20-30 人天 |

Skill 形态确实**比 MCP 少 30-40% 工作量**。但**省下来的工作量买不回上面 2 个致命问题**。

## 7. Skill 的合理用法（如果要加）

**不替换 MCP 主形态，但作为补充加分项**：

```
开发者快速接入入口：
  pip install xa-guard            ← 装 CLI / SDK
  xa-guard init-skill              ← 在项目根目录生成一个 xa-guard.skill.md
  
然后开发者用 Codex / Cline 时, AI 读取这份 SKILL.md, 
学到"在做高危操作前调用 xa-guard CLI 检查"的行为模式

本质：Skill 调本地 CLI, CLI 调真实的 XA-Guard 检测算法
```

**好处**：
- 给评委看时多一个形态（"我们覆盖 MCP + CLI + Skill 三种接入"）
- 信创场景友好（不依赖网络）
- 工程量小，作为加分项性价比高

**风险（必须告知评委）**：
- Skill 本身的 prompt injection 风险（学术已证）需要在文档里诚实承认
- 国产 IDE 覆盖率不足

---

## 8. 客观结论

| 选项 | 评估 |
|---|---|
| Skill 替换 MCP 作为主形态 | ❌ **强烈不推荐** |
| Skill 补充 MCP 作为加分项 | ⭕ 可加，但工程预算要砍掉别处 |
| Skill 完全不做 | ⭕ 也是合理选择，不影响主线 |

### 推荐方案
**默认不做**。如果 M4 阶段（提交前 1 个月）发现进度宽裕，可以**用 1-2 人天**做一个 demo 级 Skill，作为"完整性 20% 评分"的加分演示。

### 为什么不做
1. MCP 已经覆盖国产生态，Skill 反而覆盖不到
2. Skill 作为"安全工具载体"有 prompt injection 理论硬伤
3. 评委 Skill 熟悉度不够，讲解成本高
4. 工程预算应优先保证主线（MCP 6 关卡）

### 如果用户坚持要做
- 推荐：M4 阶段做"开发者快速接入 Skill"（调本地 CLI）
- 不推荐：替换 MCP 主形态、或大量工程投入
