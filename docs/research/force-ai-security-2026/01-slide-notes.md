# 现场 PPT 逐页笔记

> 说明：以下按用户提供图片顺序整理。照片存在视角、遮挡和模糊，无法辨识的文字不强行补全；扩展解释会明确写成“理解”。

## 图 1：2026 企业 AI 安全顶层设计蓝图

可辨识标题：`2026 企业 AI 安全顶层设计蓝图`

画面左侧是一个围绕 `Agent Core` 的安全结构：

- 管控流：Identity / Gateway
- 底座流：Infra Security
- 数据流：Data Isolation
- AI Resilience：撤销与恢复

右侧列出 5 个顶层设计原则：

1. 身份基建：确立智能体为第三类身份，全量接入 JIT / JEA 动态授权网关。
2. 能力隔离：严格剥离“控制流”与“数据流”，摒弃单点提示词拦截，实施可证明安全。
3. 底座防御：纠正预算错配，强制加固 AI 云底座、容器隔离与反序列化审查。
4. 人在环路：关键操作人工审批。
5. 兜底韧性：接受“系统必然犯错”前提，建立所有行动的可溯源与动作级撤销（Undo）机制。

理解：这页是整套材料的核心架构。它把智能体安全拆成“身份/网关、基础设施、数据隔离、韧性恢复”四周防护，而不是把安全能力塞在 prompt filter 一个点上。对本项目来说，这非常契合 XA-Guard 的多 Gate 架构，但还需要把“Agent 身份”和“动作级撤销”表述得更清楚。

## 图 2：OpenClaw 现象把智能体安全推成全民议题

可辨识标题：`发现二：OpenClaw 现象把「智能体安全」推成了全民议题`

左侧趋势数字：

- 6 个月内涌现 17,000+ 活跃 MCP 节点
- 4 月突破 34.6 万星标
- 72 小时突破 60,000+ GitHub 星标

右侧影响：

- 对中国 AI 产业界：证明桌面端 AIDR、Agent 身份治理、MCP 网关授权是下一代不可回避的刚需。
- 对消费市场：全民直观看见智能体代为操作文件、银行转账，“智能体安全 = 直接经济损失”成为社会共识。
- 对政策制定者：工信部与应急中心连续预警，加速了以“行动治理”为核心的监管共识落地。

理解：这页把“智能体安全”从技术圈风险变成大众可感知风险。关键变化是 Agent 不再只是聊天，它开始帮人操作文件、资金、业务系统，因此安全失误会直接变成财务、合规和运营损失。

## 图 3：攻击下沉至数据路径全链

可辨识标题：`发现三：攻击下沉至「数据路径」全链`

攻击链：

1. Malicious PDF / Slack Message
   - Hidden Instructions
   - Invisible Characters / Emojis
2. RAG Retrieval / Context Window
   - Agent processes data as trusted context
3. Toxic Combination
   - Agent triggers tool execution based on poisoned context
4. Unauthorized Execution
   - Data Exfiltration / Financial Transfer

下方要点：

- 单个工具权限合规，但组合后形成致命漏洞。
- 通过公共表单 42K 字符描述字段植入注入，绕过 CSP 外泄数据。
- 必须将“能力 × 自主性 × 权限”合并评估，采用 AIVSS 取代传统 CVSS 打分。

理解：这页最重要。传统安全看“某个工具是否危险”，而智能体安全要看“数据进入上下文后，和工具、权限、自主决策组合起来会发生什么”。也就是说，单个 MCP 工具可以合规，但 Agent 把多个工具串起来后可能越权。

## 图 4：Computer Use Agents 颠覆传统六道防线

可辨识标题：`发现四：Computer Use Agents 颠覆传统六道防线`

失效防线：

- Same-Origin Policy
- Site Isolation
- Anti-CSRF
- SameSite Cookies
- Origin Verification
- Human-in-the-Loop

突破原理：

- AI Agent 同时看到所有标签页，突破视野隔离。
- 请求均以第一方（First-party）发起，自带受害者凭证，绕过 CSRF 防御。

例子：

- RSAC 2026 标志性 PoC：通过 `file://URI` 直接外泄本地文件，过去需要恶意软件，现在只需要一个浏览器。
- Google 账号接管：刷新十次抓取 OTP。

战略要求：

- 引入可证明安全（Provable Security）架构，强制分离控制流与数据流。

理解：Computer Use Agent 把浏览器安全模型打穿了，因为它不是普通网页脚本，而是一个能“看见屏幕并操作浏览器”的自动化主体。很多 web 安全假设默认“用户是人”，但 Agent 会高频、跨页、带凭证执行动作。

## 图 5：基础设施脆弱性与预算的系统性错配

可辨识标题：`发现五：基础设施脆弱性与预算的系统性错配`

预算分配与伤害半径：

- 应用与内容层：当前预算 >60%，真实杀伤半径为单点/局部受损。
- AI 基础设施层：当前预算 <20%，真实杀伤半径为全局击穿（模型、数据、源代码、供应链全面失守）。

核心证据：

- NVIDIA Container Toolkit（CVE-2024-0132）：单 GPU 容器逃逸击穿多租户隔离。
- Hugging Face / Replicate：Pickle / GGUF 模型反序列化武器化。
- ZeroDay 基础设施：5 个 Redis 与 3 个 PostgreSQL 基础设施 RCE。

行动计划：

- 模型仓库签名验证、供应链审计必须作为强制验收项。

理解：这页强调钱和注意力投错了。很多企业把预算花在内容过滤和应用层提示词检测，但真正可能造成全局沦陷的是 AI 基础设施：容器、模型文件、反序列化、模型仓库、数据平台、供应链。

## 图 6：智能体身份独立成为第三类身份

可辨识标题：`发现六：智能体身份独立成为「第三类身份」`

左侧三类身份：

- 人类身份（Human Identity）
- 机器 / 服务账号（NHI-Static）
- 智能体身份（Agent Identity）

智能体身份特征：

- 兼具人类的广泛能力（BroadScope）与机器的指数级速度（Rapid Speed）。
- 但缺乏人类的判断力。

全新安全基线：

- JIT + JEA + JLA：彻底废除静态权限。
- Agent 的能力（Just-Enough）、范围、时长（Just-Long-Enough）必须在每次任务中按需申请（Just-in-Time）。
- AgentGateway（PEP）：统一鉴权、记录、限速与撤销的执行节点。

市场印证：

- 2026 年 5 月，Cisco 以 4 亿美元收购 NHI / Agent 身份初创公司 AstrixSecurity，标志巨头平台战正式打响。

理解：Agent 身份不是传统 service account 的小变体。服务账号通常是稳定、窄权限、可预期的；Agent 则有临时任务、动态工具、跨域数据、自然语言目标和自主规划。它需要专门的身份生命周期和授权网关。

## 图 7：影子智能体治理与 AI 韧性的崛起

可辨识标题：`发现七：影子智能体治理与「AI 韧性」的崛起`

可辨识数字：

- 80% 员工使用未经审批的 AI 工具。
- 77% 的交互涉及复制粘贴企业 PI 或专有代码。
- 额外泄露成本比标准泄露高出 `$670K`。

图示：

- Shadow AI 之上是 Agent Gateway（亮路径）。

硬塞是输家：

- 必须提供比影子路径更易用的亮路径。
- OIDC SSO、Agent Acceptable Use Policy 等治理能力是基础。

新增第三支柱（AI Resilience）：

- 无论防护多严密，Agent 必然出错。
- 防御体系从“检测 + 防护”升级为“检测 + 防护 + 韧性（UndoAI）”。
- 需要文件级行为追踪与精准撤销动作能力，而不是回滚整个系统。

理解：治理不是只靠禁用。员工会用影子 AI，是因为正式工具不好用或不够快。企业要提供更方便、更安全、更可审计的“亮路径”，同时承认 Agent 会犯错，必须设计撤销和恢复。

## 图 8：AI 安全决策的法律责任与量化估值

可辨识标题：`发现八：AI 安全决策的法律责任与量化估值`

中心公式：

```text
[Base Risk $5.9M] × [Financial Agent 1.5] × [Handles PII 1.3]
× [Autonomous 1.2] = $13.8M 潜在损失口
```

法律现实：

- 触发 10 类风险：合同违约、过失、诽谤（Walters v. OpenAI 先例）、数据合规罚款（Garante 对 OpenAI EUR 15M 罚款）、商业秘密侵占等。

董事会核心逻辑：

- 治理理投入不足该 ROI 的 10%，即具备清晰商业投资回报逻辑（Business Case）。
- 无量化合规论证（AIVSS / NIST AI RMF）将在事故后被追溯为“未尽合理注意义务”。

理解：这页把安全预算从“技术偏好”转换成“董事会可理解的风险敞口”。对比赛答辩有用：不能只说“我们能拦攻击”，还要说“我们能量化风险、保留证据、支持责任界定”。

## 图 9：企业智能体安全六大挑战

可辨识标题：`企业智能体安全六大挑战`

原因：

- 智能体具备读数据、调工具、改代码、触发流程、跨系统协同的能力。

六大挑战：

1. 权限扩张和越权：人机边界模糊，治理失效。
2. 自主执行操作，以及多智能体协同的治理和审计风险。
3. 敏感数据泄露：数据和合规主权问题。
4. 供应链安全：不安全代码进生产。
5. 提示词注入和诱导：记忆和上下文污染。
6. 内外部攻击面扩大：攻击能力放大。

底部观点：

- 更大的问题：有些企业安全团队还没有真正介入到智能体安全治理工作中。

理解：这是从企业落地角度总结的六类风险。它把提示词注入放在第五位，说明真实企业更担心权限、数据、审计、供应链和攻击面扩张。

## 图 10：企业安全处于失控边缘

可辨识标题：`企业安全处于失控边缘`

三类问题：

- 智能体安全治理问题：
  - 权限失控
  - 行为不可控
  - 身份混乱
  - 缺乏审计
- 传统安全防护能力无法应对：
  - 攻击面扩大
  - 防护滞后
  - 动态变化
  - 攻击手段进化
  - 策略失效
  - 缺乏上下文感知
- 模型安全可信问题：
  - 数据污染
  - 对抗攻击
  - 输出不可信
  - 幻觉问题

理解：这页将问题分为“治理、传统安全能力、模型可信”三层。对 XA-Guard 的表达也应分层：我们不只是模型 guardrail，而是治理中台 + 运行时管控 + 审计证据。

## 图 12：智能体安全能力层五大安全能力

可辨识标题：`智能体安全能力层 —— 五大安全能力`

五大安全能力：

1. 安全攻防
   - 模型攻防引擎
   - Prompt 防注入 / 越狱检测
   - 多轮拦截 / 记忆投毒防护
   - 数据防泄漏引擎
   - PII 数据脱敏 / 隐私保护
   - 敏感信息与凭据外泄检测
   - 自动化策略运营：链路行为追溯、日志审计、策略自动生成、回归测试
2. 行为权限
   - 资产发现与管理：智能体资产发现与分类分级、数字员工生命周期治理
   - 身份与权限管理：智能体 IAM、凭据托管、人机身份互信、动态权限
   - 运行行为管控：高危操作二次确认与拦截、异常行为分析、HITL 控制
3. 环境安全
   - 供应链与环境加固：插件工具安全扫描、白名单、沙箱隔离运行、防逃逸
   - 智能体框架安全：智能体组件 CVE 漏洞、基线
   - 智能体安全测评：提示词攻击、内容合规、安全攻防演练、漏洞评估
4. 内容安全
   - 内容合规引擎：双向内容实时合规检测、多维度检测、敏感拦截
   - 深度鉴伪：多模态伪造检测、声纹鉴别、人脸伪造识别、活体检测
   - 行业话题控制：行业话题边界控制防线、自定义模型接入专属防护
5. 审计溯源
   - 全链路日志审计：操作行为细粒度追踪记录、API 与链路调用多维度审计
   - AI 血缘与调用链：智能体及资产调用链构建
   - 安全事件分析：攻击事件还原、高危异常行为存证

理解：这页非常适合对照本仓库的 Gate1-Gate6。它提示我们，能力层应被写成可被企业安全团队理解的“攻防、权限、环境、内容、审计”五类，而不是只按实现模块命名。

## 图 13：安全智能体支撑企业安全智能化转型

  - Multi-Model Agentic Scanning Harness：多模型调度超越单一强模型。
  - 覆盖代码扫描、漏洞验证、PoC 生成、补丁创建的完整流水线。

右侧表格：

- 启示：
  - 不必等待国产 Mythos。
  - 多 Agent 天然更理解业务。
  - 前置安全左移更可落地。
- 含义：
  - 通过多智能体架构，企业完全可以用现有模型能力组合出超越单一强模型的安全红队。

- 落地方向：
  - 多 Agent 编排平台建设。
  - 业务域专属安全 Agent 开发。
  - DevSecOps 流水线集成。

理解：这页从攻击者和防守者两面看多 Agent。多 Agent 不是只增加风险，也能成为安全生产力：代码审查 Agent、合规审查 Agent、告警研判 Agent、应急处置 Agent 等。

- 安全 RAG
- 基线验证
- 已开源

理解：可信环境的重点不是“模型相信自己安全”，而是通过 TEE、远程证明、可度量启动、可信审计、IO 管控，把模型和 Agent 的运行环境变成可验证对象。

## 图 17：PrivLLM 让敏感数据可用不可见

可辨识标题：`PrivLLM：让敏感数据可用不可见`

目标：

- 让 Agent 能算，但拿不走。

三项能力：

- 协变混淆
- 可用不可见

图示大意：

- 用户信任环境中完成离线预处理和在线轻量推理。

## 图 18：PrivLLM 原理深入：LLM 的协变混淆设计

可辨识标题：`PrivLLM 原理深入 —— LLM 的协变混淆设计`

离线模型混淆：

- 乘性噪声：可相互抵消，不影响效果；有效收缩高维特征空间的攻击面。
- 加性噪声：打破明文模型与混淆模型之间的稳定数值对应关系，消除不变量。

在线混淆推理：

- 利用离线阶段的乘性词表置换，实现输入和输出数据的混淆保护。

图示：

- Embedding Layer、多个 Decoder Layer、Model Head 之间存在噪声与置换。
- 高维模型混淆支持混淆数据推理计算，同时保护能量柔性置换。
- 在线混淆数据推理保护用户数据隐私。

理解：这页比较技术细节，重点是“离线改模型 + 在线改输入输出”，使非可信推理环境中看到的是混淆后的 token / 表征，而用户侧可以解混淆。

## 图 19：PrivLLM 安全性理论框架 RmDP

可辨识标题：`PrivLLM 安全性理论框架：RmDP`

框架原理与优势：

- Rényi Metric Differential Privacy
- Rényi divergence 刻画分布关系。
- 兼容高维特征空间的高斯噪声，分析结果更准确。
- 度量空间 `(X, d)` 比较分布差异。
- 适用性更广，不局限于“相邻数据集”假设，适配灵活的语义隐私度量。

分析方法与结论：

- 离线、在线阶段独立分配隐私预算。
- RmDP 组合定理生成端到端联合隐私预算。
- 同等安全强度下，用更小噪声换来更高可用性。

理解：这页把 PrivLLM 从工程混淆提升到可证明隐私预算。对比赛不是必须实现，但可以作为“未来数据安全增强路线”的参考。

## 图 22：OWASP State of Agentic AI Security and Governance 2.01

可辨识标题：`OWASP《State of Agentic AI Security and Governance 2.01》`

副标题：

- 企业应用 AI 的梯度：从 Shadow AI 到跨组织 Agent。
- 企业到底部署到了哪一级 Agent，决定它面对风险的边界。

成熟度梯度：

- AT0 Shadow AI / 影子 AI：员工私自使用通用 AI 工具，无法治理，数据与合规风险处于隐藏状态。
- AT1 Vendor Embedded Assistant / 厂商内嵌助手：低代码 / PaaS 平台或办公软件内嵌的原生 AI 能力，功能受限、边界可控。
- AT2 Platform Integrated / 平台集成 Agent：基于企业知识库构建的问答 / 检索平台，数据本地化、权限闭环、风险集中在数据准确性与授权管理。
- AT3 Citizen Developer Agent / 低代码公民开发 Agent：连接企业内部系统与外部 API，具备基础工作流编排能力，开始涉及数据交互与接口安全风险。
- AT4 Code Executing Agent / 代码代执行 Agent：Agent 具备执行代码片段和调用核心业务 API 的能力，风险上升至业务逻辑篡改与数据泄露层面。
- AT5 Custom In-House Agent / 企业自研 Agent：深度集成核心业务流与自建模型策略，Agent 拥有极高系统权限，治理失效将导致严重业务中断。
- AT6 Externally Extended Agent (MCP) / 外部扩展 Agent：通过插件 / 协议调用第三方服务与 API，引入外部不可控因素，面临供应链攻击与合规双重挑战。
- AT7 Multi-Agent Orchestration / 多 Agent 协作网络：多 Agent 形成去中心化协作，具备自主分工与动态决策能力，行为不可预测，治理难度呈指数级上升。
- AT8 Federated / Cross Boundary / 跨组织、联邦 Agent：跨组织边界进行数据协同与联合建模，涉及多方利益与监管，是当前 AI 治理的终极挑战场景。

理解：这页给了一个非常好用的风险成熟度框架。不同 Agent 部署层级对应不同治理强度：AT0 需要发现与管控，AT3-AT6 需要身份/权限/工具/供应链治理，AT7-AT8 需要多 Agent 编排、跨边界审计和责任机制。
