## 1. 整体定位

### 1.1 核心命题

> **我们不做安全 agent，我们做"任何 agent 都能用的安全防护层"。**

这是项目最重要的产品判断。原因：

- 政企客户不会换框架（他们已经有运维工具、办公助手）
- 政企客户不会写代码（他们要的是**接入即用**）
- 政企客户要的不是"AI 工具"，而是"**让现有 AI 工具变得合规安全**"

### 1.2 类比帮助理解

类比成"电力系统"：
- ❌ **不做**：给每户家庭重建一套电路系统（=完整安全 agent）
- ✅ **要做**：在电表前装一个智能保护器（=MCP Server 防护栏）
- ✅ **同时**：定义"国家电气安全标准"（=Protocol 协议）
- ✅ **同时**：给电工提供"标准化检测套件"（=SDK 插件库 + Bench 评测）

### 1.3 4 个赛题方向的对应

| 赛题方向 | 三件套覆盖位置 |
|---|---|
| 方向 1 输入链路防御 | MCP Server 关卡 1（门口安检）+ Protocol 输入标签字段 |
| 方向 2 工具调用安全 | MCP Server 关卡 2/3/4/5 + SDK 装饰器 |
| 方向 3 插件供应链 | MCP Server 工具白名单 + AIBOM 准入网关 |
| 方向 4 评测审计 | MCP Server 关卡 6 + Bench 评测套件 + Protocol 审计日志格式 |

### 1.4 客户端兼容性原则 — 主推国产，协议中立

**MCP 协议是中立的**——遵循 MCP 协议的任何 LLM 客户端都能用我们的 XA-Guard MCP Server。这意味着我们的方案具有**真正的客户端无关性**——一次开发，全网覆盖。

**主推客户端（按本项目优先级）**：

| 客户端 | 厂商 | 类型 | 政企适配 | 备注 |
|---|---|---|---|---|
| **Trae** | 字节跳动 | AI IDE | 优 | 类 Cursor 国产代表，2025 年发布，免费，MCP 原生支持，UI 上手成本极低 |
| **CodeBuddy** | 腾讯云 | IDE 插件 / 客户端 | 优 | 腾讯生态，企业版本对接顺畅 |
| **通义灵码** | 阿里云 | IDE 插件 | 优 | 阿里生态，国企 / 央企已多有采购 |
| **Qcoder** | （以最新文档为准） | AI 编程助手 | - | 用户提及，配置方式与同类工具类似 |
| **智谱清言桌面版** | 智谱 AI | LLM 桌面助手 | 良 | 渐进式 MCP 支持 |
| **CodeGeeX** | 智谱 AI | IDE 插件 | 良 | 旧产品，MCP 支持视版本 |
| Cursor / Claude Desktop / Cline | 国外 | AI IDE / 桌面 | 一般 | **国际生态对标**，文档中作为配置参考 |

**为什么主推国产**：政企客户（政府、国企、央企）有**国产化采购合规**要求；国产工具已通过算法备案，政企采购流程顺畅。我们方案的真实落地客户用的就是上述国产工具。

**为什么仍提国际工具**：① Cursor / Claude Desktop 是 MCP 生态的代表性实现，技术参考价值大；② 部分国产工具的 MCP 支持仍在演进，国际工具的成熟实例可作为学习参考；③ 演示视频里可加 1 个 Cursor 镜头作为"国际对标"展示。

**核心产品价值**：写一次 XA-Guard MCP Server，**全部上述客户端都能用**——这是协议层产品形态的最大杠杆。

### 3.2 核心数据流（Request → Response）

一个 LLM 客户端发起工具调用，经过 XA-Guard 的完整生命周期：

```

    ↓
┌──────────────────────────────────────────────────┐
│ 关卡 1 · 门口安检                                 │
│ - 用户/工具描述提示注入检测（PromptGuard）         │
│ - 越狱模式识别（Llama Guard）                     │
```

### 3.3 关卡详细设计

#### 关卡 1 · 门口安检

**核心组件**：
- `PromptGuardClient`：调用 Meta PromptGuard 2 模型分类输入
- `LlamaGuardClient`：调用 Llama Guard 3 模型分类内容
- `ChineseFinetuner`（可选）：基于 200-500 条中文样本微调的本地模型
- `TaintLabeler`：根据输入来源（用户直接输入 / 文件 / 网络）初始化三色标签

**输入**：MCP CallToolRequest（含 tool_name, arguments, 上下文消息历史）

**输出**：
```python
@dataclass
class Gate1Result:
    decision: Literal["allow", "warn", "deny"]
    risks: list[Risk]  # 检测到的风险列表
    initial_taint: TaintLabel  # 输入数据的初始标签
    confidence: float  # 0-1
```

**性能预算**：< 200ms（本地模型推理）

#### 关卡 2 · 办事大厅

**核心组件**：
- `RiskClassifier`：根据工具元数据 + 调用参数判定风险等级
- `HITLNotifier`：通过 MCP `notifications/progress` 给客户端弹窗（Trae / CodeBuddy / Cursor 等主流客户端支持原生 elicitation；部分老版本客户端只支持 stdout 提示）
- `ApprovalStore`：审批状态持久化（避免一次会话内重复问）

**HITL 审批的 MCP 实现**：
MCP 1.0 后引入了 `elicitation`（要素征询）能力。Server 可以在 CallTool 处理过程中**反向问客户端**："这个高危操作要执行吗？" 客户端会弹窗给用户。

```python
# 伪代码示例
async def handle_call_tool(req: CallToolRequest):
    risk = classify_risk(req)
    if risk == "red":
        approval = await ctx.elicit(
            prompt="即将执行高危操作：rm -rf /var/log/*\n是否允许？",
            options=["允许", "拒绝", "永久禁止此操作"]
        )
        if approval != "允许":
            return error_response("用户拒绝")
```

#### 关卡 3 · 规则引擎

**核心组件**：
- `PolicyCompiler`：将 YAML 策略文件编译为 OPA Rego
- `OPAClient`：嵌入式 OPA 实例（pip install opa-python-client）
- `ChineseRuleBank`：30+ 条预编译的等保 2.0 / TC260 规则

**Rule YAML 示例**：
```yaml
- id: TC260-003-7.2
  name: 禁止生成涉密内容
  source: TC260-003 第 7 章
  triggers: ["content_generation"]
  predicate: |
    input.tool == "write_file"
    contains_pii(input.args.content)
  enforce: deny
  audit: required

- id: GBT-22239-8.1.4.4
  name: 重要操作授权
  source: 等保 2.0 三级 8.1.4.4
  triggers: ["exec_command", "delete_file"]
  predicate: |
    input.risk_level == "red"
  enforce: require_approval
  audit: required
```

**编译流程**（半自动）：
1. 团队人工选择关键规则
2. LLM（DeepSeek）将自然语言条款 → YAML 初稿
3. 团队 review + 写单元测试
4. 编译为 OPA Rego 入库

#### 关卡 4 · 信息流污点

**核心组件**：
- `TaintTracker`：跟踪每个数据片段的标签
- `TaintPolicy`：定义"哪种标签能流向哪种工具"

**三色标签定义**：
- `PUBLIC` (绿)：公网信息、用户直接输入
- `INTERNAL` (黄)：公司知识库、内部文档
- `CONFIDENTIAL` (红)：密码 / 密钥 / 个人信息 / 涉密文档

**传播规则**（最严格优先）：
```
PUBLIC + PUBLIC = PUBLIC
PUBLIC + INTERNAL = INTERNAL
INTERNAL + CONFIDENTIAL = CONFIDENTIAL
任意 + CONFIDENTIAL = CONFIDENTIAL
```

**工具能力声明**：
```yaml
# 工具元数据示例
tool_name: send_email
required_capabilities:
  - NETWORK_EXTERNAL  # 外网
input_max_taint: INTERNAL  # 最高接受 INTERNAL 数据
output_taint: PUBLIC  # 输出标记为 PUBLIC
```

**阻断规则**：如果工具的 `input_max_taint` 低于实际传入数据的标签 → 阻断。

#### 关卡 5 · 隔离办公间

**核心组件**：
- `SandboxRouter`：根据风险等级选择沙箱
- `DockerExecutor`：Docker + gVisor 运行高危操作
- `WasmExecutor`（可选）：WebAssembly 运行计算密集型沙箱

**配置**：
```yaml
sandbox:
  low_risk:
    type: native  # 不沙箱
  medium_risk:
    type: docker
    image: xa-guard/sandbox:latest
    network: none
    readonly_fs: true
    cpu_limit: 1.0
    memory_limit: 512m
  high_risk:
    type: docker
    runtime: runsc  # gVisor
    network: none
    readonly_fs: true
    cpu_limit: 0.5
    memory_limit: 256m
    timeout: 10s
```

#### 关卡 6 · 黑匣子

**核心组件**：
- `OTelTracer`：OpenTelemetry GenAI 语义规范
- `SMCryptoSigner`：国密 SM2 签名 + SM3 哈希
- `MerkleChain`：前向哈希链
- `TSAAnchor`：每天向时间戳服务（TSA）锚定一次

**审计日志 schema**（14 字段）：

| 字段 | 类型 | 说明 | 来源 |
|---|---|---|---|
| `trace_id` | UUID | 整次会话 ID | OTel 标准 |
| `span_id` | UUID | 单次工具调用 ID | OTel 标准 |
| `gen_ai.request.model` | str | 模型版本 | OTel GenAI |
| `gen_ai.usage.input_tokens` | int | 输入 token 数 | OTel GenAI |
| `gen_ai.tool.name` | str | 工具名 | OTel GenAI |
| `gen_ai.tool.parameters` | json | 工具参数 | OTel GenAI |
| `gen_ai.tool.result.hash` | str (SM3) | 结果哈希 | OTel GenAI |
| `gen_ai.user.role` | str | 用户角色 | **政企扩展** |
| `gen_ai.data.sensitivity_level` | enum | 三色标签 | **政企扩展** |
| `gen_ai.policy.hit_id` | list[str] | 命中的规则 ID | **政企扩展** |
| `gen_ai.tool.approval_token` | JWT | HITL 审批令牌 | **政企扩展** |
| `gen_ai.evidence.hash_prev` | str (SM3) | 前一条日志哈希 | **政企扩展** |
| `gen_ai.classify.risk_tag` | list[str] | 风险分类 | **政企扩展** |
| `gen_ai.decision.faithfulness_score` | float | CoT 忠实度 | **政企扩展** |

**法律效力**：SM3 哈希 + SM2 签名 + TSA 时间戳组合，满足《电子签名法》第 14 条"可靠电子签名"要件，可作为电子证据使用。

### 3.4 MCP 协议适配

XA-Guard MCP Server 同时扮演两个角色：

1. **对上游 LLM 客户端**：表现为一个 MCP Server（提供 tools / resources / prompts）
2. **对下游真实工具**：表现为一个 MCP Client（消费下游 MCP Servers）

这种"代理"模式技术上是 MCP Aggregator 的扩展。

**支持的 MCP 传输**：
- `stdio`（Trae / CodeBuddy / 通义灵码 / Cursor / Claude Desktop 等主流客户端默认）
- `SSE` / `HTTP`（生产部署）

**支持的 MCP 版本**：MCP 1.0+

## 4. XA-Guard Protocol

### 4.1 协议愿景

不仅是个工具，更是个**让任意 AI 智能体框架都能交换安全策略和审计证据的开放标准**。30 页规范定义：

1. **安全策略表达**（policy.yaml）
2. **审计日志格式**（基于 OTel GenAI 扩展）
3. **风险标签语义**（三色 + 多维风险标签）
4. **AIBOM 元数据格式**（CycloneDX 1.6 扩展）

## 5. XA-Guard Python SDK

### 5.1 设计目标

**给不用 MCP 的 LangChain/AutoGen 用户也能享受 XA-Guard 防护**。

### 5.2 三种集成方式

#### 5.2.1 装饰器风格

```python
from xa_guard import protect

@protect(policy="enterprise-l3")
def my_langchain_agent(query: str) -> str:
    agent = create_agent(...)
    return agent.run(query)
```

装饰器会自动：
- 在调用前跑关卡 1（输入检测）
- 在工具调用时跑关卡 3、4（策略 + 污点）
- 在结果返回时跑关卡 6（审计）

#### 5.2.2 显式 Guard 实例

```python
from xa_guard import Guard, Policy

guard = Guard(policy=Policy.from_file("enterprise-l3.yaml"))

with guard.session(user="alice", role="ops") as session:
    result = session.run(my_agent, query="...")
    audit_log = session.export_audit_log()
```

#### 5.2.3 LangChain Callback

```python
from xa_guard.integrations.langchain import XAGuardCallback

agent = AgentExecutor(
    ...,
    callbacks=[XAGuardCallback(policy="enterprise-l3")]
)
```

## 6. XA-Bench

### 6.1 评测套件愿景

> **给政企智能体出一份"中文政企版高考卷"——做出来，谁强谁弱一目了然。**

### 6.2 评测维度（7 维）

| 维度 | 用例数 | 评估方法 |
|---|---|---|
| 数据安全 | 30-50 | 投毒检测、PII 泄漏 |
| 内容安全 | 50-80 | 17 类拒答 + 31 类审查（TC260） |
| 执行安全 | 50-80 | 工具滥用、越权操作 |
| 供应链安全 | 20-30 | AIBOM 验证、依赖检查 |
| 合规风险 | 30-50 | 等保 2.0 三级要求条款 |
| 可解释 | 10-20 | CoT 忠实度 |
| 可追溯 | 10-20 | 审计日志完整性 |
| **合计** | **200-330** | |

### 6.3 用例数据来源

- **40%**：翻译/迁移 Agent-SafetyBench（清华开源中文基准）
- **20%**：网信办公开案例 + TC260 题库
- **40%**：团队基于运维场景自造

### 6.4 评测指标

| 指标 | 说明 | 算法 |
|---|---|---|
| **ASR**（Attack Success Rate） | 攻击成功率 | 直接成功 / 总测试 |
| **CuP**（Completion under Policy） | 策略约束下任务完成率 | 完成 ∧ 通过策略 / 总任务 |
| **FPR**（False Positive Rate） | 误报率 | 错误拦截 / 合法请求 |
| **Recall** | 召回率 | 检出攻击 / 总攻击 |
| **Latency P95** | 延迟（95 分位） | 测量值 |
| **Audit Completeness** | 审计完整性 | 完整审计 / 总操作 |

### 6.5 命令行接口

```bash
# 跑全套评测
xa-bench run --target http://localhost:8080 --policy enterprise-l3

# 跑某个维度
xa-bench run --dimension content-safety --target ...

# 生成报告
xa-bench report --format html  # 输出 report.html
xa-bench report --format pdf   # 输出 report.pdf
```

## 7. 部署架构

### 7.1 单机开发部署（团队开发）

```bash
# 本地 macOS / Ubuntu
pip install xa-guard-mcp
xa-guard serve --policy enterprise-l3 --port 3000
```

客户端配置（以 **Trae** 为例，其他客户端格式高度相似）：

```json
// Trae 设置 → MCP → 编辑配置文件
// 配置文件位置: Trae 设置面板里能直接打开
{
  "mcpServers": {
    "xa-guard": {
      "command": "xa-guard",
      "args": ["serve", "--upstream", "filesystem,shell"]
    }
  }
}
```

**其他客户端等价配置位置**：
- **CodeBuddy**：设置 → MCP 服务器管理
- **通义灵码**：插件设置 → 工具集成 → MCP（视版本而定）
- **Cursor**：`~/.cursor/mcp.json`（同上格式）
- **Claude Desktop**：`~/.config/Claude/claude_desktop_config.json`（同上格式）

> **关键**：JSON 配置格式在所有 MCP 客户端是一致的（MCP 协议标准）。差异只在"配置文件路径在哪、UI 怎么进入"，技术内容完全相同。

### 7.2 政企内网部署（演示用）

```yaml
# docker-compose.yml
services:
  xa-guard:
    image: xa-guard/server:latest
    ports: ["3000:3000"]
    volumes:

    environment:
      - SM_CRYPTO_KEY=/etc/xa-guard/sm2-key.pem

  opa:
    image: openpolicyagent/opa:latest
    
  langfuse:  # 审计可视化
    image: langfuse/langfuse:latest
```

## 9. 非目标（不做的事）

明确划清边界，避免范围蔓延：

- ❌ **不做完整 agent**（用户已识别，"没人会用"）

- ❌ **不做企业级 SaaS**（演示用本地部署即可）
- ❌ **不做 Kubernetes 集群**（5 个月做不完）
- ❌ **不做完整中文模型微调**（仅做 PromptGuard 中文小样本微调，不是从头训）
- ❌ **不做形式化验证**（Tool Hoare Contract 已在前期决策中砍掉）
- ❌ **不做对抗攻击实施**（防御研究，不做攻击工具）
