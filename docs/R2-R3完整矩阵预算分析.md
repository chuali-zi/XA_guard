# R2 / R3 完整矩阵预算分析（60 美元订阅约束）

> 分析日期：2026-06-22
> 依据：`D:/evidence/r2-r3-20260621b/` 的 4-job 真实 OpenCode smoke、冻结的 2,986-job plan、当前 runner 实现。
> 赛题边界：[比赛方案 PDF](./XA-202620中国雄安集团数字城市科技有限公司-面向政企场景的大模型智能体安全关键技术研究比赛方案.pdf)第 3-4 页要求可复现关键技术验证结果和量化测试效果，但没有指定 AgentDojo/InjecAgent，也没有要求 2,986-job 全矩阵。

## 结论

- 现有 `$289` 是一个有用的保守额度护栏，不是可靠的最终账单预测。
- 它来自 `4 × R2/workspace` job 的均值 `$0.09691/job` 直接乘 `2,986`。样本没有任何 R3 job，且首个 R2 job 包含一次已修复的付费失败，因此存在系统性高估；同时它也可能低估更复杂 suite、限流重试和 R3 第二阶段调用。
- 按当前逐 case runner 不改架构，完整 R2+R3 大概率仍需三位数美元；`$60` 预算仍不能保证 100% 完整矩阵。
- 比赛正式目标已升级为 `subscription_budget60_v1`：OpenCode Go 订阅可用额度按约 `$60` 管理，用于 R2/R3 双基准分层抽样；既有 smoke 和 `$10` 首批失败校准不进入新的正式统计。结果必须标记为 sampled，不能通过改 scorer、删 case 或降低门槛冒充完整验收。
- 若要把“完整矩阵”显著降价，应先做批量执行与官方日志缓存复用，再用跨 suite/R3 的小型成本校准。即使新 `$60` 订阅预算比旧 `$20` 抽样预算宽松，当前逐 case 架构仍不能保证 2,986-job 全矩阵；完整矩阵继续是研究级扩展。

## 1. 为什么是 2,986 jobs

| Benchmark | 原始 case | baseline | defended | jobs |
|---|---:|---:|---:|---:|
| R2 AgentDojo | 949 | 949 | 949 | 1,898 |
| R3 InjecAgent DS/base | 544 | 544 | 544 | 1,088 |
| 合计 | 1,493 | 1,493 | 1,493 | 2,986 |

R2 的 949 个 case 是四个 suite 的 user task × injection task 笛卡尔积：

| Suite | user tasks | injection tasks | 组合数 |
|---|---:|---:|---:|
| workspace | 40 | 14 | 560 |
| slack | 21 | 5 | 105 |
| travel | 20 | 7 | 140 |
| banking | 16 | 9 | 144 |

baseline/defended 配对不是装饰：二者必须使用相同模型、case、上游 commit 和参数，才能回答“防御降低了多少 ASR、损失了多少 utility”。只跑 defended 可以省约一半，但失去因果对照，也不再是当前验收定义下的完整矩阵。

## 2. 一个 job 为什么不等于一次模型调用

AgentDojo 是有工具交互的 agent benchmark。单个 case 会先检查 injection task 能否作为正常任务完成，再运行带注入的 user task；每段任务又可能经历多轮模型→工具→模型。因此 smoke 中每个成功 job 实际产生了 4–6 个付费 OpenCode invocation，每个 invocation 内还可能有多个计费 step。

InjecAgent 通常每 job 先调用一次模型；DS 第一阶段攻击成功时才触发第二次调用。因此 R3 的调用结构明显比 R2 便宜，不能用纯 R2 smoke 的均价替代。

费用主要来自长输入，而非输出。workspace turn 包含完整工具 schema，smoke 观察到的单 step 总 token 经常约 10k–14k，并存在缓存读取；一次 agent 轨迹又由多个 step/invocation 组成。

## 3. 4-job smoke 的真实账单拆解

从 `opencode-invocations.jsonl` 中的 provider `cost` 字段重算：

| Job | 日志成本 |
|---|---:|
| R2 workspace task0/injection0 baseline | $0.12215 |
| R2 workspace task0/injection0 defended | $0.09391 |
| R2 workspace task0/injection1 baseline | $0.06793 |
| R2 workspace task0/injection1 defended | $0.10364 |
| 合计 / 均值 | $0.38763 / $0.09691 |

首个 baseline 的第一次 benchmark attempt 在已经产生约 `$0.0632` 模型费用后，因响应 schema 不兼容而失败，第二次才成功。该兼容问题已经修复。排除这笔不应在线性预测中重复发生的一次性损耗后：

- 4 个成功结果对应成本约 `$0.3244`；
- 修正均值约 `$0.0811/job`；
- 若只对 1,898 个 R2 jobs 机械外推，约 `$154`，而不是 `$184`；
- 原 `$289` 又把同一 R2 均价套到了全部 1,088 个 R3 jobs，这一部分没有实测依据。

这里仍不能把 `$154` 当成精确 R2 报价：样本只有 workspace 的同一 user task，不覆盖另外三个 suite、不同轨迹长度、重试和限流。

## 4. `$289` 是否合理

作为批准前的最坏情况额度上限，合理：它简单、偏保守，能避免学生账户被意外跑穿；`$0.09691 × 2,986 = $289.36` 算术没有问题。

作为“完整矩阵预计实际花费”，证据不足，主要有四个问题：

1. `n=4` 太小，且四条高度相关，不是跨 suite 随机样本。
2. 样本全是 R2；R3 通常只有 1 次首阶段调用，直接套 R2 多轮 agent 均价会高估。
3. 首个 job 含一次已修复的付费失败，重复外推会高估。
4. 反方向的不确定性也存在：复杂 case、R3 第二阶段、非 JSON 重试、限流都可能增加费用。

因此当前最诚实的表述是：`$289` 是保守 cap；完整矩阵在当前架构下仍应按“三位数美元风险”管理。在没有跨 suite + R3 校准样本前，不应承诺某个更精确数字。

## 5. 当前实现中的可压缩成本

### 5.1 AgentDojo 单 case 隔离造成重复工作

当前每个 user×injection 组合都独立调用一次官方 `benchmark_suite_with_injections`。官方函数会先跑 injection-task utility；于是同一个 injection task 会为每个 user task、baseline/defended 反复计算。

四个 suite 只有 35 个唯一 injection tasks，但当前 R2 两个 arm 共触发 1,898 次 injection-utility 任务入口；若按 suite/arm 批量执行并复用官方 log cache，理论入口数可降到 70，少 1,828 次重复入口。注入后的 1,898 个核心组合仍必须真实运行，不能缓存成同一个结果。

这项优化很值得做，但不能把“入口减少 96%”误读为“总账单减少 96%”：每个 job 还包含真正的 injected user-task 轨迹。按 smoke 结构，较现实的目标是先验证 R2 成本能否下降约 30%–50%，而不是直接假定能装入 `$60` 完整矩阵。

### 5.2 避免可预防重试

- 正式跑前分别 smoke 每个 suite 和 R3，锁定 JSON/schema 稳定性。
- 将 provider/model、配置目录和 parser compatibility preflight 变成零/低成本门禁。
- 保留 resume；已经完整且 hash 匹配的 job 不重复付费。
- 失败重试应针对瞬时错误；确定性的 schema 错误应立即停止整批，而不是把 `max_attempts=2` 用满。

### 5.3 缩短适配器额外提示，但不能改 benchmark 语义

可以压缩 wrapper 自己增加的 JSON 指令、避免让模型通过额外工具读取同一 turn 文件，并验证 provider 结构化输出能力。这可能降低输入与额外 step。不能删官方 system/user/tool schema、攻击内容、parser 或 scorer；否则省下的是评测本身，不再可比。

### 5.4 换便宜模型或本地模型

这是最可能把 API 支出进一步压低甚至接近 0 的路线，但代价是：

- baseline/defended 必须从头用同一新模型重跑，不能与现有 GLM smoke 混合；
- 模型能力变化会直接改变 utility、invalid rate 和 ASR，便宜但太弱的模型可能导致验收失败；
- 本地运行把 API 费用换成 GPU/时间成本；没有合适显卡时可能非常慢；
- 仍只能声明自选模型复现，不能冒充官方排行榜成绩。

## 6. 已选定的 60 美元订阅预算方案

### `subscription_budget60_v1`：双基准分层抽样

目标是最大化竞赛答辩价值，不声称完整矩阵：

1. 现有 smoke 与 `$10` 首批失败校准保留为历史工程验证，不进入新的 sampled 分母；新预算单独计 `$60`。
2. 校准最多 `$6`：R2 四个 suite 各取 baseline/defended 配对，R3 同时取 baseline/defended，得到分 benchmark 的真实单位成本；校准样本不进入正式指标。
3. 主评测 `$48`：R2 分配 `$32`、R3 分配 `$16`。R2 各 suite 至少 8 个配对 case，其余按 suite case 数比例分配；R3 从 DS/base 固定随机抽样。
4. 重试预留 `$6`；每次调用前用保守单次预留确认“当前账本 + 下一次调用”不超过 `$60`，无法保证时提前 fail-closed。未用完的重试额度不用于越过已经冻结的样本边界。
5. 固定随机 seed `20260622`，在模型调用前冻结 calibration/sample manifest 及 SHA-256；baseline/defended 必须成对，且同模型、同 case、同上游、同参数。
6. 报告抽样分母、ASR、Utility、valid/invalid、timeout/retry、实际成本和 95% Wilson 区间，不得把 invalid/timeout 算成功。

R3 正式样本数由校准后的保守配对成本和 `$16` 配额确定，随后随 manifest 冻结。若校准显示 R2 各 suite 最低 8 对无法装入 `$32` 配额，则主评测不启动并写 `INCONCLUSIVE_BUDGET`；不得为了“跑出数字”临时降低 floor 或突破 `$60`。

按修正后的 R2 workspace 均价估算，`$60` 可支持明显大于旧 `$20` 方案的 sampled 分母，但仍需给 R3、重试和限流留出空间。最终数量必须由 `$6` 校准决定。

若 defended 样本出现 0 次攻击，约 30 个独立 valid case 才能让“零成功”的单侧 95% 上界接近 10%；一旦出现攻击，想严格证明 ASR≤10% 就需要更多样本。故点估计达标只写 `MEETS_SAMPLED_POINT_TARGET`；相应 ASR 上界与 Utility 下界也支持门槛时才加写 `CONFIDENCE_SUPPORTED`。其余情况写 `DOES_NOT_MEET_SAMPLED_TARGET` 或 `INCONCLUSIVE`。

### 分批执行要求：适配 OpenCode Go 5h/周额度

新配置默认 `max_jobs_per_invocation=8`。每次 `budget-run` 或 `budget-resume` 会先排除完整且锁匹配的结果，再从全局未完成列表取下一批；不会反复从 manifest 第一题开始。单题连续基础设施失败达到 `max_job_resume_attempts=2` 后终止该题并继续后续题，同时保持 phase 失败，避免坏题永久占用每批名额。明确的 provider 额度拒绝会暂停而非记未知成本；如果 5h 窗口或周额度接近耗尽，直接停止，等额度恢复后继续同一 phase 的 `budget-resume`。不得为了“跑完本批”突破 provider 限额、切换模型或创建新输出目录混合结果。

### 未选择方案：R3 优先完整

R3 的 544×2 jobs 通常比 R2 便宜，先做 20–40 个 R3 校准 job；只有实测完整 R3 的 90% 成本上界仍低于预算时才继续。剩余额度给 R2 四 suite 配对抽样。优点是可能完整关闭一个 blocker；缺点是未经 R3 smoke 前不能保证 `$60` 足够。

### 工程优化项：批处理/缓存

不立即花大额 API 费用。先让 AgentDojo 按 suite/arm 批量运行并复用官方 trace cache，同时保持逐 case 可审计产物、hash、resume 和原 scorer。之后用每个 suite 至少一个配对 + R3 样本重新测价。如果预测仍高于 `$60`，再选择更小抽样或便宜/本地模型。

这是长期最正确的工程路线，但它降低的是冗余，不保证仅凭当前付费模型就完成完整矩阵。

## 7. 不应采用的“降本”方式

- 不修改测试、官方数据、parser、scorer 或 ASR/Utility 门槛。
- 不只保留好结果，不删除 invalid/timeout/retry。
- 不把 baseline 和 defended 换成不同模型。
- 不把抽样结果写成 2,986-job 完整验收 PASS。
- 不把当前 4-job smoke 当成绩，也不把自选模型结果写成官方排行榜复现。

## 8. 已确认决策与执行前置

在 60 美元订阅预算和 5h/周额度限制下，不批准无分批上限的完整运行，也不批准用 `--max-jobs` 前缀冒充分层样本。执行顺序固定为：

1. 实现固定 seed 的 calibration/sample manifest、分层配额与配对校验；
2. 使用按 provider 原始 cost 的分类账和 `$60` fail-closed；
3. 离线测试与 dry-run 验证后，才使用 `$6` 校准预算；
4. 冻结正式样本数量与 hash，再按 R2 `$32` / R3 `$16` 执行，重试最多 `$6`；每次默认最多 8 jobs，按额度窗口分批 resume；
5. 报告中明确 sampled claim scope。完整矩阵只记 `research_full_matrix / DEFERRED_OPTIONAL`，不是比赛资金 blocker。

当前 runner 已实现 `subscription_budget60_v1` 所需的确定性 calibration/sample manifest、四桶 provider-cost ledger、调用前原子预留熔断、分批 resume、sampled Wilson 聚合与 artifact hash 校验，并通过离线目标测试。新的 `$6` 校准和主评测尚未运行，因此当前只能声称“执行工具已实现、正式 sampled 证据待产生”，不能声称指标达标。

在用户另行授权实现和付费前，本分析没有启动任何新模型调用，也没有修改 benchmark、测试或验收门槛。
