# SmoothLLM: 用输入扰动平滑来防 jailbreak (SmoothLLM: Defending Large Language Models Against Jailbreaking Attacks)

## 元信息
- **作者机构**: University of Pennsylvania (Alexander Robey 等)
- **年份 · 发表**: 2023-10 · arXiv (ICLR'24 / NeurIPS'24 RR 等多次引用)
- **arXiv**: https://arxiv.org/abs/2310.03684
- **本地 PDF**: ./2023-SmoothLLM.pdf
- **代码**: https://github.com/arobey1/smooth-llm
- **难度**: 3/5

## 一句话总结
把可疑输入用多种"轻微随机扰动"复制 N 份分别让模型回答，做"多数投票"——单个对抗后缀经不起扰动，平均下来 jailbreak 失败。

## 解决什么问题
2023 年中 GCG（Greedy Coordinate Gradient）攻击横空出世：在用户提问后加一段乱码（如 `! ! ! describe.\ + similarlyNow`），就能让对齐过的 ChatGPT/Llama-2-Chat 也乖乖输出违禁内容。这种"对抗后缀"攻击的特点是：
- **看起来像乱码**，没语义
- **极度精细**：稍微改一两个字符就失效

之前的 jailbreak 防御都是"训练阶段"的（RLHF、安全微调），但 GCG 这种"测试时对抗"很难训出免疫。SmoothLLM 想找一个**测试时**就能跑、对各种 jailbreak 攻击都通用的轻量防御。

## 用了什么方法
**核心打比方**：你怀疑面前的鸡尾酒有毒，不敢直接喝。你把它倒进 10 个杯子，每杯加一点不同的稀释液（水、果汁、可乐），再分别小口尝。如果有 7 杯说"没问题"、3 杯说"有怪味"，按多数投票判定"基本安全"——而对抗扰动通常经不起这种稀释。

**核心三步**：
1. **生成 N 份扰动副本**：对原始输入做 N 次（如 N=10）轻微随机扰动，扰动方式有三种：
   - **Insert**：随机位置插入随机字符
   - **Swap**：随机交换两个字符位置
   - **Patch**：随机区间替换为随机字符
   - 扰动比例 q（如 10%）控制改动量
2. **N 次独立推理**：把 N 个扰动版本分别喂给 LLM，得到 N 个回答
3. **多数投票**：用一个"是否违规"分类器判断每个回答；如果**多数回答违规**，认定攻击成功并拦截；否则按某个非违规回答输出

**与之前方法的区别**：
- vs **训练时防御 (RLHF, safety SFT)**：SmoothLLM 是测试时，不动模型
- vs **Llama Guard / 分类器**：SmoothLLM 不靠分类器找"恶意意图"，而是利用对抗输入的"脆弱性"
- vs **EraseCheck**：EraseCheck 是删除-检验，SmoothLLM 是扰动-平滑——思路相似但实现不同

## 为什么能解决
关键直觉：**GCG 这类优化攻击的对抗后缀对扰动极度敏感**。改一个字符就失效，因为它本质是"利用模型的某个奇异点"找到的精细扰动。而正常 prompt 加几个字符不会改变语义——模型回答仍然合理。

用统计语言说：jailbreak 攻击的成功率函数在"输入空间"里有一个极窄的"漏洞窗口"；扰动让输入大概率落到窗口外，攻击就失败了。

**何时会失效**：
1. 攻击者**针对 SmoothLLM 自适应优化**——优化扰动鲁棒的对抗后缀（确实有后续工作做到这点）
2. 扰动比例太小，攻击仍然挺住；太大，正常 query 的语义也被破坏
3. N 次推理成本 = N 倍 LLM 调用

## 主要结果
- 在 GCG attack + Vicuna / Llama-2-7B-Chat 上：
  - 基线 ASR ~ 70%，SmoothLLM 后降至 **~5%**（约 14 倍降幅）
- 对 PAIR、AutoDAN 等其他 jailbreak 攻击也有效，ASR 普遍降 70%+
- 对正常 query 的拒绝率（"安全税"）：约 +5%（轻微误拦）
- 推理成本：N 倍（典型 N=10，10 倍）

## 局限性
1. **N 倍推理成本**显著，对成本敏感场景不友好
2. 对**自适应攻击**（论文末尾自己讨论了）效果会下降
3. 扰动破坏正常长输入的语义（如代码、JSON），可能降低正常任务质量
4. 后续研究表明 SmoothLLM 可被 robust GCG / AutoDAN 部分绕过

## 我们项目里的用法
**对应关卡**：第 1 关「输入安检门」+ 第 6 关「输出审查」的可选增强项。
- **是否采用**：建议作为**高安全场景**的可选模式，不作为主线（成本太高）
- **核心借鉴**：SmoothLLM 揭示的"对抗后缀经不起扰动"是个**关键洞察**——可以更轻量地实现：只做 N=2 或 N=3 的快速对比，看回答的稳定性
- **演示价值**：可以做一个"敏感操作时启用 SmoothLLM"的演示分支，展示对 GCG 类强攻击的防御能力
- **结合**：与 PARDEN（让模型复读）思路相近，可考虑融合——既复读又扰动
