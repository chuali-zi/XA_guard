## 一句话总结
用结构因果模型(SCM)给 Agent 决策做"反事实审计", 回答"如果当时输入变一变, 它会怎么决定"。

## 这篇论文研究什么
"为什么 Agent 做了这个决定" —— 这是政企场景最难回答的问题。监管者要求企业解释"为什么 AI 拒绝了 X 用户的贷款申请", 公司只能说"模型说不应通过", 监管者必然追问"如果 X 的收入再高 1 万, 会通过吗?"。

之前不足:
- 解释性方法(SHAP, LIME)针对静态分类器, 对多步 Agent 不适用
- LLM 的 CoT 看似是"解释", 但研究表明 CoT 不一定忠实反映真实推理(见 Faithful-CoT 论文)
- 缺乏"反事实"分析框架: 改一个输入看决策怎么变

希腊神话中, 阿里阿德涅给特修斯一团线, 让他走出迷宫。本项目名喻意: 给 Agent 决策迷宫一团"因果线", 沿线追溯就能找到出口。

## 它提出了什么方法
作者把 Judea Pearl 的结构因果模型(SCM)框架引入 Agent 审计:

**SCM 建模**:
- 把 Agent 决策的关键变量画成因果图
- 节点: 用户输入、RAG 检索结果、工具返回值、LLM 中间状态、最终决策
- 边: 因果依赖(包括混淆变量)

**三层因果分析(Pearl 阶梯)**:
1. **关联(Association)**: P(决策|输入), 普通统计
2. **干预(Intervention)**: P(决策|do(输入=X)), 主动改变输入看影响
3. **反事实(Counterfactual)**: P(决策 | 输入=X, 已观测到决策=Y), "如果当时是 X 而不是观测到的输入, 决策还会是 Y 吗"

**审计管线**:
1. 录制 Agent 决策的完整状态(类似 Replayable Agent)
2. 在录制基础上做反事实回放: 改一个变量, 重跑 Agent
3. 输出"反事实因果图": 哪些输入是决定性的, 哪些是无关的

**关键工程问题**: LLM 不可干预 —— 你不能直接"do" 模型内部权重。本文用代理方法:
- 用 patching(把模型某一步替换为另一步)做近似干预
- 用 prompt perturbation 测试因果敏感性
- 用多次采样估计反事实分布

打个比方: 这是给 Agent 决策做"如果当时...会怎样"的事后复盘工具, 让审计员能像法庭重审证据一样, 对每个决策反复推敲假设场景。

## 为什么这个方法有效
关键设计哲学: 真正的"可解释"需要因果而非相关 —— Pearl 50 年来在因果推断的工作给了理论基础。失效场景: SCM 建模需要先验知识构造因果图, 对开放领域 Agent 难以完整建模; 反事实回放计算开销大。

## 主要实验结果
- 在金融贷款 Agent 和医疗咨询 Agent 上做实验
- 反事实审计可识别 ~80% 的"决定性输入"
- 与人工审计师一致性 ~75%, 优于 CoT 自解释的 ~40%
- 计算开销: 每次审计需要 50-200 次额外的 Agent 推理

## 局限性
1. SCM 图需要专家构造, 不能自动学习
2. 计算开销大, 不适合实时
3. 对"涌现"行为(Agent 突然偏离预期)难以建模

## 我们项目里的用法
**关卡 6(黑匣子审计) + 政企"呈堂证据"场景**: 反事实审计是黑匣子的"高级形态" —— 不只录像, 还能"假设性回放"。建议:
1. 对所有政企关键决策(审批、拒绝、放行)采用反事实审计
2. 输出"决策敏感性报告": 哪些输入是关键的, 改一点点就会换决策
3. 这种报告在监管层面极有说服力, 远比"LLM 给了这个答案"更可信
4. 计算开销大, 不必每次都做, 但对争议案例必做

## 学习路径
1. 必看 Judea Pearl 的《The Book of Why》前 3 章, 理解 SCM 和反事实
2. 看 Ariadne 论文 §3 SCM 形式化 + §4 LLM 干预方法
3. 在我们的 Replayable Agent 基础上叠加反事实回放接口
