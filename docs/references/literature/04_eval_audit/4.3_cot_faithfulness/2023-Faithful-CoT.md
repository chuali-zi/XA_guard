# 忠实思维链推理 (Faithful Chain-of-Thought Reasoning)

## 元信息
- **作者机构**: University of Pennsylvania, Allen AI, Microsoft (Qing Lyu, Shreya Havaldar 等)
- **年份 · 发表**: 2023 · IJCNLP-AACL 2023
- **arXiv**: https://arxiv.org/abs/2301.13379
- **本地 PDF**: ./2023-Faithful-CoT.pdf
- **代码或项目主页**: https://github.com/veronica320/Faithful-COT
- **难度**: 4 颗星

## 一句话总结
传统 CoT 是"看起来对", 本文让 CoT 是"真的对" —— 把推理交给可验证的符号求解器。

## 这篇论文研究什么
CoT 让 LLM 准确率大幅提升, 但有个隐患: LLM 写的 CoT 推理过程, **不一定**真实反映它内部如何得出答案。可能出现:
- 推理步骤逻辑错误, 但最终答案对(蒙对了)
- 推理步骤逻辑对, 但最终答案错(算错了最后一步)
- 推理步骤是事后编造的合理化(post-hoc rationalization)

这对审计是致命的: 如果 CoT 不忠实, 那它作为"AI 解释"就是误导性的。监管要解释, AI 给一段花言巧语 —— 看着合理, 其实是骗局。

## 它提出了什么方法
作者的解决方案极聪明: **把"推理"和"算"分开**。

**两阶段管线**:
1. **翻译阶段(LLM 负责)**: 把自然语言问题翻译成"程序" + "自然语言注释交错"
   - 例: 问题"小明买 3 个苹果每个 5 元, 又买 2 个橙子每个 8 元, 共多少钱"
   - LLM 输出: 
     ```
     # 苹果总价
     apple = 3 * 5  
     # 橙子总价  
     orange = 2 * 8  
     # 总和
     total = apple + orange
     ```
2. **求解阶段(确定性求解器负责)**: 把生成的程序丢给 Python / SymPy / 数据库去执行
   - 输出: total = 31

**关键性质: 设计上的忠实性(Faithful by Construction)**
- 自然语言注释是"推理叙述"
- 程序代码是"推理执行"
- 两者一一对应(LLM 在生成时就保证)
- 执行结果由确定性求解器给出, 不依赖 LLM
- 因此: 解释(注释)和答案(求解器输出)绝对对齐, 不可能出现"嘴上说一套, 心里算另一套"

**应用领域**:
- 数学推理(Python 求解)
- 多跳问答(SQL 查询)
- 时间推理(逻辑求解器)
- 规划任务(PDDL 求解器)

打个比方: 传统 CoT 是"让学生口述解题思路, 然后报答案" —— 思路和答案可能不一致; 本方法是"让学生写出标准格式的解题过程, 老师用计算器逐步验证" —— 思路 = 答案, 不可能撒谎。

## 为什么这个方法有效
设计上分离"言"和"行", 把"言"作为"行"的注释, 保证两者一致。失效场景: 不适合无法形式化的任务(开放式问答、创意写作); 翻译错误时不可知。

## 主要实验结果
- 在 7 个推理基准上测试(数学、问答、规划、时间)
- 比标准 CoT 平均提升 ~10 个百分点
- 100% 忠实性保证(执行结果即注释承诺)
- 在 GSM8K: 81% → 86%, 同时给出可验证的程序

## 局限性
1. 仅适合可形式化任务
2. 翻译质量受 LLM 能力限制
3. 求解器需要预先准备, 不是通用方案

## 我们项目里的用法
**关卡 3(规则编译器) + 关卡 6(黑匣子审计)**: 这篇论文给出"可信解释"的工程蓝本。具体建议:
1. 关卡 3 的规则编译器, 应当让 LLM 把业务规则翻译成可执行 DSL(像 Rego 或 OPA), 由确定性引擎执行 —— 而非 LLM 直接判断
2. 关卡 6 黑匣子日志应包含"程序 + 自然语言注释"双轨, 监管可以验证 LLM 给出的解释与实际执行完全一致
3. 这是政企场景的"呈堂证据"金标准: 不是"AI 说这样判", 而是"AI 写了程序 X, 程序 X 按规则 Y 执行, 输出 Z"
