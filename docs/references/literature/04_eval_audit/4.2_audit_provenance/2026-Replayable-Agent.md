## 一句话总结
为金融智能体设计"录像 + 回放 + 重审"机制, 让每一笔交易决策都能事后复盘。

## 这篇论文研究什么
金融场景对智能体最特殊的要求是"可复盘": 一笔交易出问题, 监管会要求金融机构在 30 天内提交完整的决策依据, 包括当时 AI 看到的数据、用了什么模型、上下文窗口里有哪些信息、做了什么决定。但 LLM 智能体天生有"不可复现性":
- 模型权重随时被更新(GPT-4o → GPT-4o-2024-11)
- 上下文窗口的内容(RAG 检索结果)依赖检索时点
- 温度采样(temperature > 0)每次输出不同
- 工具返回值依赖外部 API 状态

之前不足:
- 大多数 Agent 框架只记录"输入输出文本", 不记录上下文构造细节
- 模型版本、随机种子、检索快照通常无法回滚
- "回放"实际上做不到逐字节复现, 只是"差不多重跑一次"

## 它提出了什么方法
作者把智能体当成"可回放虚拟机"来设计:

**核心架构: Event Sourcing + 快照**

1. **事件日志层**: 每一步决策记录:
   - LLM 输入快照(完整 prompt, 含 system + user + history + tool returns)
   - LLM 输出快照(原始 logits 前 K 名 + 采样结果)
   - 工具调用 trace(参数、返回值、时间戳)
   - RAG 检索快照(检索 query + 返回的 chunks + 嵌入向量哈希)
   - 模型版本指纹(model name, weight hash, sampling temperature, seed)

2. **快照层**: 关键时点(交易完成、风险事件)对整个状态打快照

3. **回放器**:
   - 给定一个时间点, 从快照恢复 + 重放事件日志
   - 支持"完全确定性回放"(锁定 seed + 用记录的 token 而非重采样)
   - 支持"反事实回放"(改一个输入, 看决策变不变)

**整体流程**: 像数据库的 WAL(Write-Ahead Log)—— 每次写之前先记日志, 数据库崩溃后能从日志恢复; Agent 决策出问题后, 从事件流恢复到任意时点重审。

打个比方: 这是给智能体装了"飞机黑匣子" —— 不光记录"飞机什么时候坠毁", 还能从黑匣子还原"坠毁前 30 分钟所有仪表盘读数 + 飞行员操作", 让事故调查能逐秒重演。

## 为什么这个方法有效
事件溯源是已被数据库行业验证 30 年的成熟范式, 把它迁移到 Agent 上是工程合理的。失效场景: 外部 API 不可幂等(比如调用真实股票交易 API)无法真正回放; 上下文窗口太大时日志体积爆炸。

## 主要实验结果
- 实测金融顾问 Agent, 100,000 次交易决策全部可完整回放
- 完全确定性回放成功率 ~98%(剩 2% 因外部 API 不一致)
- 日志体积约为输入文本的 3-5 倍
- 反事实回放可定位 ~85% 的决策偏差归因

## 局限性
1. 日志体积大, 长期存储成本高
2. 外部 API 不幂等问题没有彻底解决, 只能"近似回放"
3. 主要针对金融场景, 其他领域需要重新设计事件粒度

## 我们项目里的用法
**关卡 6(黑匣子审计)**: 这篇论文几乎就是关卡 6 的"教科书蓝本"。建议直接套用其设计:
1. 每次 Agent 决策记录 6 个维度: prompt 快照、输出 logits、工具 trace、RAG 快照、模型版本、随机种子
2. 关键事件(签字、转账、外发文件)打快照
3. 提供回放 API, 监管/审计员可以"重演任意一次决策"
4. 政企场景对"呈堂证据"要求更高, 建议每条事件用前一条的哈希链起来(类似区块链), 防篡改
