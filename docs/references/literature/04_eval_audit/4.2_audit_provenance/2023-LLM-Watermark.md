# 大语言模型水印技术 (A Watermark for Large Language Models)

## 元信息
- **作者机构**: University of Maryland (John Kirchenbauer, Jonas Geiping 等)
- **年份 · 发表**: 2023 · ICML 2023 Outstanding Paper
- **arXiv**: https://arxiv.org/abs/2301.10226
- **本地 PDF**: ./2023-LLM-Watermark.pdf
- **代码或项目主页**: https://github.com/jwkirchenbauer/lm-watermarking
- **难度**: 4 颗星

## 一句话总结
让 LLM 在生成时悄悄"打上指纹", 检测时只需统计词频就能识别"这段话是 AI 写的"。

## 这篇论文研究什么
ChatGPT 出来后, 一个老问题变得迫切: 怎么判断一段文字是人写的还是 AI 写的?学生用 AI 作弊、新闻媒体用 AI 造假、攻击者用 AI 批量生成钓鱼邮件 —— 监管者和教育界都急需一个"AI 内容检测器"。

之前的检测方法有两类:
- **训练分类器**(OpenAI 自己出过): 准确率不够高, 容易被改写攻破
- **零样本统计**(GLTR): 看 token 概率分布, 有用但不可靠

**核心问题**: 检测是"事后"的, 不依赖原始模型, 很难做到既高准又抗改写。

## 它提出了什么方法
作者提出"主动埋点" —— 在生成时就把指纹种下去:

**核心想法(简单到优雅)**:
1. 把模型的词表(vocabulary)按某个种子随机分成"绿色 list"和"红色 list"
2. 每次生成下一个 token 时, 对"绿色 list"的 token 给一个 logit 加分(比如 +2)
3. 模型在不影响流畅度的前提下, 倾向于多选绿色 token
4. 检测时: 用同样的种子重算"绿色 list", 统计文本中绿色 token 比例, 用 Z 检验判断

**伪随机的关键**:
- 种子来自前一个 token 的哈希值
- 这样不需要任何"密钥服务器", 任何持有种子函数的人都能检测

**整体流程图**(打比方):
- 生成 = 餐馆做菜时悄悄在每道菜里放微量"我的私章"调料
- 检测 = 食客拿菜化验, 看私章调料的浓度是否显著高于自然均值
- 私章调料无毒、无味, 客人吃不出来, 但化验仪器一测就知道是哪家餐馆做的

**理论保证**:
- 提供严格的假设检验, 给出"误报率"上界
- 对短文本(25 token)也能达到 99.9% 检测准确率
- 对改写攻击有一定鲁棒性(改写率 < 50% 时仍能检测)

## 为什么这个方法有效
关键设计哲学: 把"是否 AI 生成"问题转化为"统计假设检验" —— 数学上可证明的低误报率, 而不是 ML 模型的"差不多"。失效场景: 极短文本(< 25 token)无法检测; 对手知道算法后可针对性改写绕过; 多次重写后水印强度衰减。

## 主要实验结果
- 200 token 文本: 99.99% 检测准确率, 误报率 < 10^-6
- 25 token 极短文本: 仍能达 99% 检测准确率
- 困惑度仅上升 ~5%, 人眼几乎看不出生成质量下降
- 对 token 替换攻击, 替换 25% 仍能检测; 替换 50% 失败

## 局限性
1. 必须在生成时埋点, 对模型 API 提供商有依赖
2. 对手知道算法可针对性改写
3. 多语言、多模态(代码、JSON)上效果未充分验证

## 我们项目里的用法
**关卡 6(黑匣子审计)**: 这是水印的核心应用场景。我们的智能体生成文本时(回执、文书、邮件), 应当对每段输出都打上水印, 这样事后只要拿到任何"嫌疑文本", 都能验证它是否出自我们的智能体。在政企场景下, 这相当于给每份 AI 生成内容盖上"防伪章", 对追责、维权、合规审计都至关重要。具体实施: 在 LLM 调用层(LiteLLM/vLLM)注入 watermark logits processor, 检测器作为黑匣子审计模块的子组件。
