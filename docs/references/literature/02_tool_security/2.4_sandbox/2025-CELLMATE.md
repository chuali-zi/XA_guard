## 一句话总结
浏览器智能体(Claude Computer Use 类)的 HTTP 流量拦截沙箱,限制网络出入站、敏感字段、cookie。

## 解决什么问题
浏览器智能体(Browser-Use、Claude Computer Use、ChatGPT Atlas、OpenAI Operator)是 2025 年最火的应用方向之一。但它们有独特的高风险:① **同源策略失效**——智能体可以"看见"用户所有网页内容,绕过浏览器原本的隔离;② **登录态扩散**——智能体读到任一标签页的 cookie,可能用它访问其他敏感站点;③ **真实操作不可逆**——下单、付款、删数据,一次错就完蛋。已有 sandbox 方案(gVisor、Firecracker)是进程/虚机级的,**不适配浏览器这种"特定于 HTTP 协议"的细粒度安全需求**。CELLMATE 专门为浏览器智能体场景设计了一套 HTTP 流量级别的沙箱机制。

## 用了什么方法
CELLMATE 的核心机制:

1. **HTTP Interceptor(流量拦截层)**:
   - 部署在浏览器和远端服务器之间(本地代理或浏览器扩展)。
   - 拦截**所有出入站 HTTP/HTTPS 流量**,对内容做细粒度检查。
   - 类似 Burp Suite 但是为智能体场景优化。

2. **策略层(Policy Layer)**:
   - 定义对每个域的访问能力:`allowed_domains`(白名单)、`required_auth`(必须有的 header)、`forbidden_fields`(不可发出的字段如信用卡号)、`max_data_size`(单次请求大小上限)。
   - 策略可按"任务"、"用户"、"敏感级别"分层应用。
   - 可结合 LLM 生成 policy(Conseca 思路)——根据当前任务即时生成。

3. **敏感数据掩码(Sensitive Field Masking)**:
   - 用 NER + 正则识别请求/响应中的 PII(身份证、银行卡、Email、密码等)。
   - 出站时按策略决定:`block(拦截)/mask(脱敏后放行)/log(记录)`。
   - 入站时也可掩码——防止页面里的敏感信息被智能体读到。

4. **多标签页隔离**:
   - 每个智能体任务在独立的浏览器 profile(独立 cookie 容器)中运行。
   - 任务结束销毁 profile,避免数据残留。
   - 跨标签页通信需经过 policy 审核(类似 IsolateGPT 的 Hub-Spoke)。

5. **审计日志**:
   - 完整记录所有 HTTP 流量(可选哈希链 + 国密签名,与方向 4 衔接)。
   - 异常流量自动告警。

类比:这就是**"为浏览器智能体的 mitmproxy + WAF + DLP"** 三合一——既能拦截可疑请求,又能脱敏敏感数据,还能审计完整流量。

## 为什么能解决
传统 sandbox 是"边界内一切都允许,边界外一切都禁止"——粒度太粗。HTTP 级别拦截能做"按内容、按字段、按域名"的细粒度控制,这才是浏览器智能体的真实需求空间。同时,浏览器智能体的所有操作都最终落到 HTTP——拦截 HTTP 就是拦截所有动作。这种"在抽象边界做检查"是软件安全的成熟思路(中间盒、API gateway、service mesh)。

## 主要结果
- 对 5 类浏览器智能体场景(电商、社交、企业 OA、搜索、邮件)做攻击与防护评测。
- 防护后,9 种已知攻击(数据外发、跨域偷 cookie、未授权付款等)拦截率 **96%**。
- 性能开销:平均请求延迟 +30-80ms(可接受)。
- 正常任务完成率损失 ~6%(主要是部分敏感字段被掩码导致工具调用失败需要重试)。
- 多任务并发隔离效果良好——独立 profile 设计避免了跨任务污染。

## 局限性
1. **TLS 拦截需要中间人证书**——智能体环境得信任 CELLMATE 的 CA,部署门槛。
2. 对 WebSocket、HTTP/3 等非标准流量支持有限。
3. PII 识别基于规则+NLP,新型敏感数据(如内部代号)可能漏判。
4. 不防客户端 JavaScript 执行后的影响(那是浏览器内核的事)。

## 我们项目里的用法
**对应关卡 5(沙箱)**。如果我们的运维助手需要做"网页操作"(查内网监控页、改配置后台),可以借鉴:
- ① **HTTP 拦截**——用 mitmproxy 起一个本地代理,所有智能体的 HTTP 流量经过它,做策略拦截。
- ② **政企特化**:在 PII 字段上加上"身份证/工号/政务系统 ID/涉密标识"等中文场景实体。
- ③ **接关卡 6(审计)**:HTTP 流量审计自然天成——与方向 4 的哈希链审计直接对接。
- ④ **答辩亮点**:"我们的沙箱对网络流量做语义级别拦截,而不仅是进程隔离"——这能区别于一般 gVisor/Firecracker 方案。
- ⑤ **MVP 选择**:即使我们的运维助手主要是命令行,也可以保留一个简化的 HTTP 拦截层,用于"任何工具想发外部请求都要过 policy"。
