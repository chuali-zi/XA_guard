# 评估智能体环境中的自动化提示注入攻击(Assessing Automated Prompt Injection Attacks in Agentic Environments)

## 元信息
- **作者机构**: (2025) / 多机构
- **年份 · 发表**: 2025 · arXiv 预印本
- **arXiv**: https://arxiv.org/abs/2606.10525
- **本地 PDF**: 无
- **难度**: 3 星

## 一句话总结
系统评估"用一个自动化流程（而非人工）在动态智能体环境里生成并投递提示注入"能达到什么效果，为"自动红队"作为方法论正名并给出评估口径。

## 解决什么问题
提示注入研究大量停留在人工构造样本。当攻击进入**智能体动态环境**（多轮工具调用、状态演化），需要回答：自动化生成的攻击在这种环境里有多有效？怎么公平评估？本文把"自动化 + 智能体环境"这两个变量一起纳入评估。

## 用了什么方法
1. **自动化攻击流水线**：程序化生成注入、投递到智能体执行路径、判定成功。
2. **动态环境评估**：在多步工具调用的智能体环境里测，而非单轮静态。
3. **标准指标**：ASR 为主，兼顾正常任务完成度。

## 为什么能解决
把"自动化"当作一等公民来评估，明确了自动红队相对人工的成本/覆盖优势与效果边界，也暴露了"自动生成质量参差、需要好的成功判定器"这一工程要点。

## 主要结果
- 自动化攻击在智能体动态环境中可达到可观 ASR。
- 强调成功判定器（oracle）质量对评估可信度的决定性影响。

## 局限性
1. 自动生成的 payload 质量方差大，需筛选。
2. 判定器噪声直接影响结论。
3. 环境覆盖仍受限于所用基准。

## 我们项目里的用法
**为整套 conductor 的"自动生成 + 自动评估"闭环提供方法论背书。** ① 印证我们把"云端 agent 自动造 payload + OAR 自动判定"作为核心循环的合理性；② 强调判定器质量——我们不自造 oracle，而是复用 OAR 已有的 **ledger + property/oracle + accountability** 作为高可信成功判定，规避本文点名的"判定器噪声"风险；③ `evaluator.py` 的胜负判定严格以 OAR verdict 为准，不让 LLM 自评。

## 学习路径
- 看自动化流水线各环节 —— 对照 conductor 的 SEED→RUN→EVALUATE。
- 看判定器讨论 —— 理解我们为何坚持用 OAR ledger 判定而非 LLM 自评。

→ **映射到本工作流**：conductor "自动生成+自动评估"闭环的方法论依据；`evaluator.py` 坚持 ledger 判定。
