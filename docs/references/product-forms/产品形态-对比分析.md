## 0. 一句话结论

> **维持 MCP Server 三件套作为主形态。Skill / CLI / SaaS 都不能作为替代。CLI 和 SDK 已在三件套中合理覆盖；Skill 仅可作为 M4 加分项（1-2 人天）；SaaS 完全不做。**

---

## 1. 四种形态的本质区别

| 形态 | 用户怎么用 | 谁主动 | 部署在哪 |
|---|---|---|---|
| **MCP Server**（主推） | 改一行 `mcp.json` 配置 | **客户端自动调用**（透明拦截） | 本地进程 / 内网 |
| **Skill** | 在对话里"用 xa-guard 检查" | **AI 主动读 SKILL.md** | 项目目录 |
| **CLI** | `xa-guard scan ./log` | **人手动跑** | 本地命令行 |
| **SDK / Library** | `from xa_guard import guard` | **开发者代码里 import** | 应用进程内 |
| **外置 SaaS** | `POST https://api.../check` | **应用代码 HTTP 调用** | 远程云端 |

**关键洞察**：
- MCP Server / SaaS = **AI 控制层独立于 agent 执行循环**（Forrester 2025-12 "Agent Control Plane" 概念）
- Skill / CLI / SDK = **嵌入到 agent 执行循环内部**

业界趋势：**前者会胜出**（Snyk 2025 / Forrester 2025-12 / Deloitte 2025 一致判断）。

## 3. 各形态的"硬伤"（对项目而言）

### Skill 的硬伤
1. **prompt injection 载体悖论**：arXiv 2510.26328 + ClawHavoc 事件（1200 恶意 Skill）证明 SKILL.md 本身是攻击载体。**用"已知有安全短板的载体"做"安全工具"——双战场**
2. **国产 IDE 全员不支持**：Trae / 通义灵码 / 文心快码截至 2026-05 都没原生支持 SKILL.md
3. **不能做长时监控**：Skill 是"配方"，没有持久态

### CLI 的硬伤
1. **AI 是实时交互**，离线 CLI 价值低
2. **业界没有"CLI 主形态"的运行时安全产品**（Garak/PyRIT 都是评测工具）
3. **政企不为单纯 CLI 买单**

### SDK 的硬伤
1. **同质化严重**：LlamaFirewall + Guardrails AI + NeMo Guardrails 已占位
2. **不能覆盖 MCP 客户端用户**（Trae / CodeBuddy / 通义灵码用户没法装我们 SDK）
3. **政企采购买"平台"不买"SDK"**：2025Q1 中标 1.24 亿，没有纯 SDK 中标

### 外置 SaaS 的硬伤
1. **政企不允许**：60% 企业要求本地化，央国企近 100% 拒绝
2. **商业市场无纯 SaaS 赢家**：Lakera / CalypsoAI / Robust Intelligence / Protect AI 全部被收购
3. **学生团队不能承担 24x7 运维**：演示当天云挂 = 0 分

---

## 4. MCP Server 的独特优势（为什么是"最优解"而非"已选"）

MCP Server 不只是"我们选了它"，而是**多个维度同时最优**：

### 4.1 政企采购最契合
- 协议中立 + 本地部署 + 国产生态全覆盖
- 类比"统一安全网关"概念，政企 IT 团队心理模型友好

### 4.2 创新差异化最强
- **中文政企 Policy DSL 编译器**：在 MCP Server 形态下作为可视化卖点最炫
- **三色信息流污点**：在中间件层执行，效果可视化
- **国密哈希链审计**：在 Proxy 层做审计最干净

这些创新点**在 SDK 形态下会退化**（变成普通 validator），**在 Skill 形态下无法实现**（Skill 无持久态）。

### 4.3 演示视频效果最佳
- Trae（中文 IDE）基础 MCP / fallback + 支持 elicitation 的客户端弹窗审批 + 实时拦截 → 评委 1 分钟懂价值
- 其他形态都需要更多解释

### 4.4 业界趋势顺势
- Forrester 2025-12 "Agent Control Plane" 概念
- Snyk 2025 判断"复刻 Web 安全 WAF 路线"
- Lakera 等商业产品演进表明 Proxy 形态是终局

## 5. 用户原始直觉的客观评估

### 5.1 用户原话拆解

> "做一个安全的 agent 没用 因为大家不会用你的 agent，要做的应该是个安全的插件类的"
> "起一个 skill + cli 安全外置服务也是个不错的选择"

### 5.2 直觉里**正确的部分**

| 用户洞察 | 评价 | 体现在 |
|---|---|---|
| "不做完整 agent，要做插件" | **完全正确** ✅ | 这正是 MCP Server 形态的本质 |
| "安全外置服务" | **完全正确** ✅ | MCP Server 就是"通过 MCP 协议的外置服务" |
| "因为 mcp 是通的" | **完全正确** ✅ | 协议中立性是 MCP 的核心价值 |
| 关注信创/本地化场景 | **完全正确** ✅ | CLI 形态在信创场景确实有用 |

### 5.3 直觉里**不准确的部分**

| 用户假设 | 实际情况 |
|---|---|
| "Skill + CLI" 可能比 MCP 好 | ❌ Skill 国产 IDE 不支持；CLI 不能做运行时防护 |
| 把"安全外置服务"和 MCP 当两件事 | ❌ MCP Server 就是一种"安全外置服务"（用 MCP 协议） |
| "起一个新形态"避免随大流 | ❌ 业界趋势恰恰证明 MCP/Proxy 形态是对的，不是随大流 |

### 5.4 综合评估

**用户的产品直觉是非常好的**（"做插件而不是 agent"、"安全外置服务"两个核心洞察都对）。

**但具体方案选择上**，**MCP Server 比 Skill+CLI 更好地实现了用户直觉里的核心理念**——这反而是个"好消息"：你的直觉没错，只是用了不同词汇描述同一件事。

## 6. 如果硬要折衷：可行的"三层覆盖"方案

如果团队就是想"多形态覆盖"，**最大化兼容**用户直觉，可以这样做：

```
            ┌─────────────────────────┐
   用户/AI →│  Skill (M4 加分项)       │ ← 项目目录里的 xa-guard.skill.md
            │  让 AI 知道"调 CLI 检查" │   开发者快速接入入口
            └────────┬────────────────┘
                     ↓ Skill 指示 AI 调 CLI
            ┌─────────────────────────┐
            │  CLI (内置在 XA-Bench)   │ ← xa-guard scan / xa-bench run
            │  评测、信创本地化场景    │   命令行入口
            └────────┬────────────────┘
                     ↓ CLI 内部调用核心算法
            ┌─────────────────────────┐
            │  MCP Server (主形态)     │ ← 同一份策略 + 检测算法
            │  运行时实时防护          │   核心
            └────────┬────────────────┘
                     ↓ 多产品共用
            ┌─────────────────────────┐
            │  策略规则 + 检测算法     │ ← 单一事实源
            └─────────────────────────┘
```

**这是 R1 调研报告建议的方案**。它的核心是：**MCP Server 仍是主形态**，CLI 和 Skill 只是"开发者接入入口"——三者共用同一份策略和算法。

**工程量估算**：
- MCP Server（已规划）：核心 4 个月
- CLI（XA-Bench 已规划）：0 额外工作
- Skill（M4 加分项）：1-2 人天

**对评委的叙事价值**：
> "我们覆盖政企现有 AI 工具链所有主流接入方式——MCP（Trae / CodeBuddy / 通义灵码用户）+ CLI（信创私有化场景）+ Skill（开发者快速接入）。三种形态共用同一份策略规则与检测算法。"

这个叙事**确实加分**（"完整性 20%"加 3-5 分）。但**不要拿来替换主线**。

## 7. 最终推荐 + 行动建议

### 7.1 最终推荐

**Tier 1 必做（主线）**：
- ✅ MCP Server（关卡 1-6 + AIBOM 准入）
- ✅ Protocol（30 页规范）
- ✅ SDK（LangChain 适配）
- ✅ XA-Bench（评测套件，天然 CLI 形态）

**Tier 2 可加（M4 加分项）**：
- ⭕ Skill（1-2 人天，开发者快速接入入口，调 CLI）
- ⭕ SDK 留 `mode="remote"` 配置（0 工程量，仅留白）

**Tier 3 不做**：
- ❌ 单独的 CLI 主形态（XA-Bench 已覆盖）
- ❌ 任何形式的 24x7 SaaS 服务

### 7.2 工程预算分配

总预算：5 个月 × 2-3 人 = 10-15 人月

| 项目 | 预算 | 备注 |
|---|---|---|
| MCP Server（6 关卡） | 6-8 人月 | 主线 |
| XA-Bench（200-300 用例 + CLI） | 1.5 人月 | 评测 |
| Protocol（30 页规范） | 1 人月 | 与 MCP 并行 |
| Python SDK | 0.5 人月 | M5 做 |
| Skill 加分项 | **0.05 人月（1-2 人天）** | M4 加 |
| 演示前端 + 视频 | 1 人月 | 必做 |
| 缓冲 / 调试 | 1-2 人月 | |

**关键**：Skill 只能用 **0.05 人月**（1-2 人天），不能多。

### 7.3 答辩问答弹药

| 评委可能问 | 我们的回答 |
|---|---|
| "你们为什么用 MCP 不用 SaaS？" | 政企 60% 要本地化 + 央国企近 100% 拒绝 SaaS；商业 SaaS 全被收购；学术也不走 SaaS 路线 |
| "为什么不用 Skill？" | Skill 是 prompt injection 载体（arXiv 2510.26328）；国产 IDE 不支持 SKILL.md；不胜任长时监控 |
| "为什么不做 SDK？" | 同质化严重（已有 LlamaFirewall / Guardrails AI / NeMo Guardrails）；政企不买单 SDK |
| "MCP 在国内能落地吗？" | Trae / CodeBuddy / 通义灵码全部原生支持，国产生态成熟 |
| "如果客户用 LangChain 不用 MCP 呢？" | 我们的 SDK 作为补充覆盖此场景，三件套设计已经考虑 |
| "未来 SaaS 化怎么办？" | SDK 已留 `mode="remote"` 配置接口（如果做了） |

## 10. 给用户的客观回应（**最重要的章节**）

> 用户原话："多子agent进行web然后你再做research，然后出几个平行的docs，然后你来跟我讨论讨论利弊，可行性(你必须客观，我只是个随机想到的点子，但是不一定正确，不要被我带偏)"

### 10.1 我的客观判断

**Skill + CLI 安全外置服务" 这个组合方案不如现有 MCP 三件套**。具体理由 6 条：

1. **Skill 在国产生态覆盖率几乎为零**（Trae / 通义灵码 / 文心快码全不支持），与"主推国产"叙事直接冲突
2. **Skill 作为"安全工具载体"有学术证实的 prompt injection 硬伤**（arXiv 2510.26328 + ClawHavoc 事件）
3. **纯 CLI 不能做运行时防护**，业界没有"CLI 主形态"的运行时安全产品
4. **创新点（中文政企 Policy DSL）在 Skill/CLI 形态下会退化失锚**
5. **"安全外置服务"概念用 MCP 协议实现比自定义 HTTP API 好**（蹭生态 + 协议中立）
6. **MCP 已是 2025-2026 业界共识**（Forrester / Snyk 等独立分析一致判断）

### 10.2 但用户洞察里有 4 个**完全正确**的部分

1. ✅ **"做插件不做 agent"**：MCP Server 正是这个本质
2. ✅ **"安全外置服务"**：MCP Server 就是
3. ✅ **协议互通**：MCP 协议中立性
4. ✅ **关注信创/CLI 场景**：CLI 可以作为信创备胎（XA-Bench 已覆盖）

### 10.3 综合结论

**新点子方向是对的，但 MCP 已经实现了所有正确的部分**。增加 Skill 形态作为加分项可以做（1-2 人天），但**不要重做产品形态**。
