# 目标与范围

## 总目标

建设一个真实、重型、可持续扩展的企业智能体安全靶场，用于红队演练、漏洞验证、能力检测、自动化评测和证据化审计。

靶场必须能回答：

1. 智能体是否会被复杂输入链路攻击诱导。
2. 工具调用是否会越权、泄密或造成未授权副作用。
3. 插件、Skill、脚本和依赖是否存在供应链风险。
4. 防护系统是否能给出可验证、可复盘、可追责的审计证据。

## P0 范围：重型可演示靶场

P0 不做轻量 demo，而是做一个足够沉的最小企业靶场。

P0 必须包含：

| 项 | 目标 |
|---|---|
| 企业域 | 办公协同、运维生产、研发供应链、治理审计。 |
| 角色 | 普通员工、运维、研发、HR、财务、部门负责人、安全审计、红队。 |
| Agent | 办公助手、运维助手、研发助手、数据助手、安全审计助手。 |
| 工具面 | 15-20 个 MCP-like tool 或业务 API 模拟工具。 |
| 攻击 case | 至少 30 个。 |
| benign control | 至少 30 个。 |
| assurance check | 至少 10 个。 |
| 多步链 | 至少 3 条。 |
| 供应链链路 | 至少 1 条。 |
| 审计篡改链 | 至少 1 条。 |
| 审批绕过链 | 至少 1 条。 |
| RAG/文档注入链 | 至少 1 条。 |
| 冒充高管付款链（BEC） | 至少 1 条。 |

P0 输出：

1. 靶场设计文档。
2. 初始场景矩阵。
3. 独立 case schema。
4. 证据规范。
5. 后续 runtime 实施计划。

## P1 范围：企业完整靶场

P1 面向持续红队和回归测试。

P1 目标：

| 项 | 目标 |
|---|---|
| 企业域 | 5-6 个完整业务域。 |
| 工具 | 50+ tools。 |
| case | 100+ attack case、100+ benign control。 |
| 多 Agent | 支持委托链和跨 Agent 权限传递。 |
| 协议 | 支持 stdio、HTTP、模拟 IDE 客户端。 |
| 供应链 | 插件市场、AIBOM、制品 hash、CI/CD。 |
| 沙箱 | 支持禁网、只读、资源限制验证。 |
| 报告 | 生成机器可读 JSON 和人类可读 HTML/Markdown。 |

## P2 范围：研究级/决赛级靶场

P2 面向高阶展示和研究。

P2 目标：

1. 多租户企业。
2. Shadow AI 发现模拟。
3. Agent 身份生命周期。
4. JIT/JEA/JLA 权限签发。
5. 风险金额量化。
6. Undo / 补偿动作建议。
7. 大规模自动化 red-team runner。
8. 外部 benchmark 与内部企业靶场融合。
9. 第三方 TSA/HSM 证据链对接。
10. 攻防演练大屏和复盘报告。

## 明确不在 P0 做的事

1. 不实现真实生产连接器。
2. 不实现真实攻击外部目标。
3. 不做恶意软件执行环境。
4. 不使用真实个人数据。
5. 不把本地模拟证据宣传为生产验证。
6. 不把靶场代码写入根 `src/`。
7. 不把靶场文档塞入既有 `docs/`。

## SUT 范围

`SUT` 是 System Under Test，即外部被测防护系统。它可以是：

1. XA-Guard。
2. 其他 MCP 安全代理。
3. 原生 Agent runtime。
4. 自研工具审批系统。
5. 空防护 baseline。

靶场只定义输入、环境、期望、证据和指标，不绑定任何 SUT 内部实现。
