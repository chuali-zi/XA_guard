# 设计说明

## 一句话定位

Enterprise Agent Range 是一个真实企业级智能体安全靶场，用于检验 Agent Gateway、防护栏、MCP 安全代理、审计系统和红队策略在复杂业务环境中的真实有效性。

它不是轻量 demo、不是普通 CTF、不是单点 prompt injection 数据集，而是一个有组织、有业务、有权限、有敏感数据、有工具链、有审批、有审计、有供应链和有多步攻击链的企业数字孪生环境。

## 设计目标

1. 真实模拟企业环境。
2. 支持红队持续攻击和漏洞复现。
3. 支持防护系统能力检测。
4. 支持自动化指标评测。
5. 支持证据链和审计复盘。
6. 支持多 Agent、MCP 工具、RAG、插件供应链和审批链组合测试。
7. 与任何具体防护产品解耦，把被测系统统一视为 `SUT`。

## 为什么必须重型

轻量 prompt injection 测试只能证明模型或规则是否识别某句话，无法证明智能体运行时是否安全。企业智能体风险往往发生在组合链路里：

- 文档中隐藏指令污染上下文。
- Agent 读取内部数据后调用外部工具。
- 低权限用户诱导高权限 Agent 执行动作。
- 插件声明无害但安装脚本执行外联。
- 审批被伪造、重放或绕过。
- 审计记录存在但无法还原真实行为。

因此靶场必须包含业务上下文、身份、数据域、工具能力、审批、审计和供应链，而不是只跑单条字符串。

## 设计原则

### 企业真实感优先

场景应像一个真的数字城市科技集团，而不是抽象玩具系统。系统之间要有历史包袱、权限差异、误配置、人工流程和业务压力。

### 攻防闭环优先

每个攻击 case 必须能映射到防护能力、oracle、证据和复现步骤。不能只记录”可能存在风险”。

### Mock 系统不预置防护

业务系统和工具的 mock 实现本身不做安全拦截判断；凡是”应不应该被允许”的判断只存在于 case 的 `expected`/oracle 里，由 SUT 去做决策。如果 mock 工具自己先拦截了危险操作（例如把 `exec_command` 改成只认白名单命令），Null Adapter（无防护基线）就会呈现出虚假的安全表现，无法反映 SUT 的真实增量价值。靶场要模拟的是企业里”该出错就会出错”的真实脆弱面，不是先把自己修成一个安全样板；角色、部门、审批链等组织设定可以尽量精细真实，但精细化的目的是让 oracle 判断更准确，不是把防护逻辑塞进 mock 系统本身。

### 可复现优先

所有自动化 case 必须确定性、安全、可重复执行。随机、时间、外部依赖和人工步骤都要显式记录。

### 副作用可控

破坏性动作必须进入 mock sink、沙箱或可回滚资源。禁止真实删除、真实外发、真实付款、真实生产变更。

### 解耦优先

靶场不依赖任何单一防护产品内部实现。XA-Guard、其他 Agent Gateway、原生 MCP client 都应可作为外部 `SUT` 接入。

### 证据优先

输出必须包括 case 输入、期望、实际结果、下游副作用、审计记录、hash、环境信息和版本信息。

## 设计中的核心对象

| 对象 | 说明 |
|---|---|
| Enterprise | 模拟企业组织和业务上下文。 |
| Principal | 人类员工、外包、红队、审计员。 |
| Agent | 办公、运维、研发、数据、安全审计等智能体身份。 |
| Tool | MCP 工具或业务 API 封装。 |
| Data Asset | 文档、日志、记录、密钥样本、PII 样本、业务单据。 |
| Policy Expectation | 靶场自己的期望行为，不引用任何 SUT 内部规则。 |
| Scenario | 一组有业务目标的多步任务。 |
| Test Case | 可执行或可人工验证的最小验证单元。 |
| Evidence | 执行后的原始证据和派生报告。 |

## 靶场成功标准

P0 成功不等于实现所有企业系统，而是要达到：

1. 有完整设计文档。
2. 有独立运行边界。
3. 有清晰场景矩阵。
4. 有可执行 case schema。
5. 有明确指标和 oracle。
6. 有证据规范。
7. 后续实现不会污染或耦合 XA-Guard 主产品代码。
