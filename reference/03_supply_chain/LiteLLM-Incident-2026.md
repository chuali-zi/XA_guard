# LiteLLM 2026 年 3 月 PyPI 包污染事件案例分析

> **声明**：本案例的具体技术细节我们应在自己写报告时进一步核实最新公开来源。本文件基于业界已公开的概述编写，作为团队答辩与方案撰写的**典型案例参考**。

## 基本信息
- **事件时间**: 2026 年 3 月
- **影响主体**: LiteLLM（开源 LLM 代理网关项目，pip 安装包名 `litellm`）
- **事件性质**: PyPI 包供应链污染（具体引入方式以官方公告为准）
- **披露来源**: OWASP Top 10 for LLM v2025 文档及多家安全媒体报道
- **重要性**: ★★★★★ 这是迄今为止**最有冲击力的**"为什么要做 AI 供应链审计"的真实案例

## 事件梗概

LiteLLM 是一个被广泛使用的 LLM 代理网关项目（GitHub 高星，pip 月下载量很大），它的作用是**给 100+ 家 LLM 厂商（OpenAI、Anthropic、Google、阿里、智谱……）的 API 提供一个统一调用接口**——开发者不用为每家 API 写一套适配，直接用 LiteLLM 即可。

由于其在 AI 生态中是"基础设施"地位，被嵌入到大量 AI 流水线和企业级 LLM 应用中（包括很多企业内部产品）。

2026 年 3 月，LiteLLM 的某个 PyPI 发布版本被发现存在**供应链层面的问题**，**在公开披露的报告中，业界给出的影响数字是"约 40 分钟内波及 40,000+ AI 流水线"**——这也是 OWASP Top 10 for LLM v2025 在 LLM03（供应链）章节引用此事件作为典型范例的核心数据。

## 为什么这是关键案例

对我们项目（雄安智能体安全比赛）来说，这个事件提供了：

### 1. 强论据：现状不安全
传统 SBOM/PyPI 信誉模型对 AI 工具链的覆盖不足：
- 一个被广泛信任的中间件（LiteLLM）出问题，会通过 API 代理形态**横向波及**所有用它接 LLM 的产品
- 影响是**指数级**的：影响 ≠ 受影响的人数，而是受影响的人数 × 每个产品的下游用户

### 2. 强论据：AI 供应链特性独特
- 传统 npm/PyPI 包仅传播一次（你装了就在你机器里）
- LLM 代理工具一旦被污染，**每一次调用 LLM 都会被中间人**——这种"长尾持续中间人"是 AI 时代特有
- 上游 LLM 厂商也无法察觉（它看到的就是"一个正常客户端"）

### 3. 强论据：监管必然加严
事件之后，**OWASP Top 10 for LLM v2025 直接把 LLM03（供应链漏洞）从 v1 的位置往前提到第 3 位**——业界共识就是要把这块管起来。
