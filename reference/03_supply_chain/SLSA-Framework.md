# SLSA 软件供应链可信级别框架（Supply-chain Levels for Software Artifacts）

## 基本信息
- **类型**: 业界供应链可信级别框架
- **维护机构**: Open Source Security Foundation（OpenSSF, Linux Foundation 旗下）
- **最新版本**: SLSA v1.0（2023 年达 1.0 稳定）
- **官方链接**: https://slsa.dev
- **是否强制（政企场景）**: 推荐

## 一句话总结
告诉你"从源代码到发布版本"这条供应链每个环节应该做到什么级别。

## 这是什么

SLSA 起源于 Google 内部的 "Binary Authorization for Borg"，2021 年开源。它不是 SBOM——SBOM 是**结果**（一份成分表），SLSA 是**过程**（生产过程的可信度）。

SLSA 把供应链可信度分成 **L1/L2/L3/L4** 四个等级（v1.0 简化为 L1/L2/L3）：
- **L1**: 自动化构建 + 有出处声明（最低门槛）
- **L2**: 出处声明被签名 + 托管构建服务
- **L3**: 构建环境强隔离 + 出处声明完全可追溯

## 关键能力

1. **出处声明（Provenance）**：每个发布版本都要附带"我是怎么被构建出来的"声明
2. **构建隔离**：高级别要求构建过程在隔离环境进行，防止开发者绕过流程
3. **签名验证**：出处声明被签名，可以独立验证
4. **威胁建模对齐**：明确说明"做到 L3 能挡掉哪些已知攻击模式"

## 我们项目里的用法

学习思路而非直接落地——我们项目本身不涉及 SLSA 完整实施。但**可以在 30 页方案的"合规对齐"章节提一句**："我们的 AIBOM 准入网关在判定第三方插件可信度时参考 SLSA 等级，要求至少 L2"，这给评委传达一个信息：「我们对接业界最佳实践」。

## 学习建议

- **必看**：https://slsa.dev 主页 + Levels 介绍页（5 分钟可读完）
- **跳过**：完整 spec、threat model 长文档（这些是给做 CI/CD 平台的人看的）
- **答辩可用一句话**：「SLSA 三级要求构建环境强隔离，对应到 AI 工具链就是模型训练全程不能被开发者篡改」
