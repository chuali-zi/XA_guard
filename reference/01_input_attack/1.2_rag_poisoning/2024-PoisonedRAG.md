# PoisonedRAG: 给 RAG 知识库下毒的攻击 (PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation)

## 元信息
- **作者机构**: Pennsylvania State University + Illinois Institute of Technology (Wei Zou, Jinyuan Jia 等)
- **年份 · 发表**: 2024 · USENIX Security 2024
- **arXiv**: https://arxiv.org/abs/2402.07867
- **本地 PDF**: ./2024-PoisonedRAG.pdf
- **代码**: https://github.com/sleeepeer/PoisonedRAG
- **难度**: 3/5

## 一句话总结
首次系统化研究"给 RAG 知识库塞几篇恶意文档"就能让 LLM 在特定问题上回答攻击者预设的错误答案，注入 5 篇文档攻击成功率高达 90%。

## 解决什么问题
RAG（检索增强生成）是当前企业级 LLM 应用的主流架构：把企业内部文档/知识库做成向量库，用户问问题时先检索 Top-K 相关文档，再喂给 LLM 生成答案。这带来一个新的攻击面——**知识库不可信**。

之前业界关注的多是"prompt injection"（攻击用户输入），但 RAG 让攻击面扩展到了"document injection"（攻击企业知识库）。如果攻击者能往知识库里塞文档（如通过工单上传、公开文档抓取、内网钓鱼），就能：
- 让客服机器人回答错误的产品信息
- 让运维助手按攻击者意图执行操作
- 让医疗问答系统给出错误建议

这篇论文是**第一篇**系统化研究 RAG 投毒攻击的工作。它做的是攻击不是防御，但**对防御侧极其重要**——只有理解攻击长什么样，才能设计有效防御。

## 用了什么方法
**核心打比方**：图书馆来了个坏读者，往哲学书架塞了 5 本伪造的"哲学经典"，里面写着"康德最爱吃汉堡"。读者查"康德的爱好"时，那 5 本书会被找出来，他自然就被骗了。

**攻击者两步走**：
1. **检索条件**：恶意文档必须能被检索器命中目标问题 → 文档里要包含目标问题的关键词、语义近似
2. **生成条件**：恶意文档被检索后，LLM 必须真的输出攻击者想要的答案 → 文档里要写得"煞有介事"，让 LLM 信以为真

**两种威胁模型**：
- **黑盒**：攻击者不知道检索器和 LLM 细节，靠 GPT-4 生成"看起来权威 + 包含目标关键词 + 断言错误事实"的伪造文档
- **白盒**：攻击者知道检索器 embedding 模型，用梯度优化方法生成对抗后缀（类似 GCG），最大化与目标 query 的余弦相似度

**攻击优化为一个数学问题**：找到最小数量的恶意文档，使得 (a) 被检索器选中 Top-K + (b) LLM 在这些文档辅助下输出目标答案。

## 为什么能解决（这里是"为什么攻击能成功"）
关键直觉：**LLM 在 RAG 里有一种"语义信任"——它会优先相信被检索回来的文档而不是自己的参数化知识**。这是 RAG 设计的初衷（克服幻觉、加入新知识），但也让恶意文档成了高效攻击载体。攻击者不需要做白盒优化，靠 GPT-4 写几段"伪权威文档"就能成功。

**何时会失效**：
1. 知识库本身有严格的访问控制（攻击者无法写入）—— 但很多 RAG 系统就是用互联网抓取的数据
2. 检索器对污染文档天然不友好（如恶意文档与正常文档分布差异巨大）
3. 防御方案如 RobustRAG、TrustRAG 已能显著降低 ASR

## 主要结果
- 注入 **5 篇恶意文档**到百万级知识库，对 GPT-3.5 / GPT-4 / Llama 系列攻击成功率均达 **~90%**
- 黑盒攻击与白盒效果相近 → 攻击门槛极低
- 对 LangChain、LlamaIndex 等主流 RAG 实现普遍有效
- 攻击文档可通用化——一个"康德爱吃汉堡"模板，可生成成百上千个目标问题的攻击样本

## 局限性
1. 攻击者需要能"写入"知识库（实际部署中并非总能）
2. 论文测试主要用维基百科/NQ 等公开数据集，企业内部知识库的实际效果可能不同
3. 多文档时如何不被检索器"稀释"是个 trade-off

## 我们项目里的用法（防御视角）
**对应关卡**：第 3 关「RAG 知识库防污染」+ 第 7 关「审计溯源」。
- **作为威胁模型基线**：我们的演示场景必须包含 PoisonedRAG 攻击 demo——这是最经典、最有说服力的 RAG 安全事件。把 PoisonedRAG 的攻击脚本跑通，作为我们防御方案要对抗的基准
- **数据合成**：用 PoisonedRAG 的方法生成 100-200 条中文政企场景的恶意文档（伪造的政策解读、伪造的运维流程文档），作为评测集
- **防御对应**：第 3 关用 TrustRAG / RobustRAG 思路（cluster filter + isolate-aggregate），目标是把 PoisonedRAG 攻击的 ASR 从 90% 降到 < 15%
- **溯源连接**：与方向 4 审计联动——攻击发生后用 Traceback-RAG 思路定位污染文档（哪批文档被投毒、入库时间、上传用户）
