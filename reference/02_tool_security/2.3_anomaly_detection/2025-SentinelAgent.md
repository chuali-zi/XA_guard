# SentinelAgent:多智能体系统中基于图的异常检测(SentinelAgent: Graph-based Anomaly Detection in Multi-Agent Systems)

## 元信息
- **作者机构**: arXiv 2505.24201 作者团队(2025 年 5 月预印)
- **年份 · 发表**: 2025 · arXiv 预印
- **arXiv**: https://arxiv.org/abs/2505.24201
- **本地 PDF**: ./2025-SentinelAgent.pdf
- **代码**: 论文中注明的 repo
- **难度**: 3 星

## 一句话总结
把多智能体执行链建模为图,用图神经网络检测异常的节点/边/子图,提早发现"链路被劫持"。

## 解决什么问题
多智能体协作场景(coding agent + reviewer agent + executor agent;或者 RAG agent + retrieval agent + writer agent)在政企应用越来越常见。但有个特殊风险:**整条链路上任意一个 agent 被注入或被换成恶意 agent,整个任务结果就被劫持**。而且因为 agent 之间是黑盒调用,**单独审查任一 agent 都看不出问题**——只有把整条链路放到一起看,才能发现"这个 reviewer agent 突然给一段恶意代码盖章通过"这种异常。SentinelAgent 提出把整个多智能体执行序列建成**任务执行图**,用图分析方法做异常检测。

## 用了什么方法
SentinelAgent 的核心步骤:

1. **任务执行图建模**:
   - **节点**:每个 agent 的一次调用(包含 agent 名、输入 hash、输出 hash、调用时间戳)。
   - **边**:agent 之间的消息传递(包含消息内容 embedding、消息类型)。
   - 整个任务执行过程构成一个有向图,可能涉及环(同一 agent 多次被调用)。

2. **特征工程**:
   - 节点特征:agent 的角色 embedding、输入/输出长度、与历史调用的相似度、决策耗时等。
   - 边特征:消息 embedding、消息和上下文的语义一致性分数。
   - 全局特征:总耗时、agent 切换频率、消息总长度等。

3. **图神经网络(GNN)检测器**:
   - 用 GCN 或 GAT 做表示学习——每个节点学到一个综合 embedding。
   - 训练目标:把正常执行图和异常执行图(人工注入攻击 + 真实漏洞数据)做二分类。
   - 异常发现可定位到具体节点/边——"这个 agent 的输出和它的角色明显不一致"。

4. **在线/离线两种模式**:
   - 离线:对完整执行图做异常打分,适合审计。
   - 在线:边执行边构建子图,对当前路径打分超阈值则告警(给 HITL 看)。

类比:这就是"图谱版的入侵检测系统(IDS)"——传统 IDS 看网络流量序列,这里看 agent 调用序列;传统 IDS 用 Snort 规则,这里用 GNN 模式识别。

## 为什么能解决
关键洞察:**单 agent 的输出表面合法,但放在执行链上下文看就违和**。例如 reviewer agent 通常会指出代码 bug,某次突然全盘 OK 通过——单独看没问题,放到历史轨迹中看就是 outlier。GNN 善于捕捉这种"图结构 + 节点属性"的复合异常。这是把多年来网络入侵检测、欺诈检测、社区异常发现的成熟方法迁移到 LLM 智能体场景。

## 主要结果
- 构造了 5 个多 agent 任务环境(SWE-bench, HotpotQA, 多 agent 写作等)。
- 在每个环境中注入 4 类攻击:agent 替换、prompt 注入、协作干扰、输出篡改。
- SentinelAgent 检测 F1 = 0.84,baseline(单 agent 异常检测)F1 = 0.51——图视角带来显著提升。
- 检测延迟 < 200ms(对每次新 agent 调用)。
- 在大规模(20+ agents)场景下,GNN 训练需要 ~10 万条样本——数据需求大。

## 局限性
1. 需要预先训练 GNN,**数据冷启动问题严重**——新场景下没有足够数据训。
2. 对全新攻击模式(GNN 训练数据没见过的)检测率低。
3. 多 agent 系统结构变化时(加入新 agent、改变协作模式)需要重新训练。
4. GNN 解释性较弱——能给出"这里有异常",但难给出"为什么"。

## 我们项目里的用法
**对应关卡 6(审计) + 关卡 4(评测)**。SentinelAgent 是我们审计层的**异常检测引擎候选**:
- ① 我们的运维助手如果是多 agent 架构(规划 agent + 执行 agent + 审计 agent),可借鉴 SentinelAgent 的图建模方法。
- ② **简化复现**:不用从零训 GNN,可以用更简单的"规则 + 统计"基线——记录每个 agent 的"行为分布",新输入超出 3σ 就告警。等数据多了再升级到 GNN。
- ③ **答辩亮点**:"我们提出多 agent 执行链异常检测器,在简化模型下能检测出 80% 的注入攻击"。
- ④ 配合方向 4 审计——异常告警自动写入哈希链日志,作为后续溯源证据。
