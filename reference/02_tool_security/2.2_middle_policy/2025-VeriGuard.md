## 一句话总结
不止"运行时检查"规则,而是用形式化验证证明智能体在所有可能执行路径上都满足安全政策。

## 解决什么问题
所有运行时防护(AgentSpec、ShieldAgent、GuardAgent)都有一个共同 weakness:**它们只检查"已发生的动作"——攻击者可能找到一条它们没覆盖的路径绕过去**。要真正"可证明"安全,需要在系统部署前就对所有可能的执行路径做形式化分析,确认所有路径都符合政策——这就是软件验证(formal verification)的经典思路,在飞控、内核、密码学库上已被使用多年。VeriGuard 把这套方法搬到 LLM 智能体上。

## 用了什么方法
VeriGuard 的形式化框架(以下是高层描述,完整理解需要看论文):

1. **智能体的形式化建模**:
   - 把 LLM 智能体的行为抽象为一个**状态转移系统(STS)**——状态 = (memory, tool_state, conversation_state),转移 = LLM 决策产生的下一动作。
   - LLM 自身的不可预测性用**非确定性选择**建模——任何可能的 LLM 输出都纳入考虑空间。

2. **政策的形式化表达**:
   - 用 **Linear Temporal Logic (LTL)** 或 **Computational Tree Logic (CTL)** 表达政策。
   - 例:`G (send_email → ¬contains_sensitive)` 表示"任何时刻,只要发邮件,内容必须不含敏感信息"。

3. **验证流程**:
   - 用模型检查器(model checker,如 NuSMV、Spin)对 STS × LTL 公式做穷举检查。
   - 如果有反例(违反政策的执行路径)——输出该路径供分析,通过修改政策或加入约束消解。
   - 如果无反例——产生一份机器可验证的证明书。

4. **可执行运行时**:
   - 通过验证的 policy 可编译为运行时 monitor(类似 AgentSpec 的执行引擎)。
   - 这样既有"事前可证明"又有"运行时可拦截"。

类比:VeriGuard 像"飞机起飞前的全套体检"——不是飞起来才发现问题,而是用数学证明"在任何天气、任何乘客载重组合下都不会失稳"。

## 为什么能解决
形式化验证在传统软件已被证明能给出"绝对保证"(seL4 微内核、CompCert 编译器、各种密码学库)。把它应用到 LLM 智能体的核心挑战是:**LLM 输出空间太大,如何抽象化?** VeriGuard 的关键是把 LLM 输出抽象为 **(action, args) 离散动作**——这样不需要对 LLM 内部建模,只对它的可能输出建模,大幅简化验证。这是工程妥协(我们不验证 LLM 内部,只验证"如果 LLM 输出任意 action,系统是否安全"),但实际可执行。

## 主要结果
(以下基于论文实验,具体数字待论文最终版)
- 在 AgentDojo 上构建形式化模型,验证 12 条政策约束。
- 通过验证的 policy 在运行时拦截 ASR 接近 0%(因为是数学证明覆盖所有路径)。
- 验证成本:小型模型(5-10 个工具)能在分钟级完成验证;大型(50+ 工具)需要小时级。
- 自动从自然语言 policy 提取 LTL 公式的准确率约 65%(其余需要专家手改)。

## 局限性
1. **状态空间爆炸**——工具数/上下文复杂度增大时验证不可行。
2. LLM 抽象为离散动作可能丢失语义(比如同一动作不同参数可能效果差很大)。
3. 自然语言到 LTL 的编译仍依赖 LLM,可能漏判隐式约束。
4. **工程门槛极高**——团队需要形式化方法专家;直接复现不容易。

## 我们项目里的用法
**对应关卡 3(规则编译)的最高境界**。我们**不建议完整复现**(工程量大、专业门槛高、收益边际),但可以借鉴:
- ① **作为答辩"未来工作"**部分提及——展示我们对"可证明安全"这一未来方向的认知和路线规划。
- ② 对**少数关键场景**(如涉密数据处理)做轻量形式化验证——可用 Z3 SAT solver 验证 2-3 条核心 invariant,而不是全套 LTL+model checking。
- ③ 与 ShieldAgent 形成互补论述:ShieldAgent 是"概率性安全保证(MLN 软规则)",VeriGuard 是"绝对性安全保证(model checking)"——按场景风险等级选择不同强度。
- ④ 学习意义大于工程价值——团队成员通过精读 VeriGuard 加深对智能体安全研究前沿的理解。
